## Problem Statement

It's hard to keep track of who you talked to and what each conversation was
about. Existing tools cover pieces — Granola does meeting notes, Google Calendar
holds meetings, ChatGPT can answer questions about pasted text, a notes app
holds follow-ups — but the loop never closes. You end each week with notes
scattered across four surfaces and no single place that says "here's what's on
your plate today and who you're seeing."

## What Makes This Cool

A single command (`/brief`) in any Claude Code session that pulls today's
calendar events, looks up your past Granola notes for each invitee via Granola's
MCP server, and returns one formatted brief: who you're meeting today, what you
last talked about, and what you said you'd follow up on. No daemon, no database,
no web app — just a SKILL.md prompt template that orchestrates two existing
data sources through an agent that's already on your machine.

The unlock is the realization that the "tool" isn't a backend, it's a prompt.
Granola has an MCP server (Feb 2026). Your calendar is one CLI call away. A
gstack skill is the smallest possible artifact that stitches them.

## Constraints

- Personal use only — no multi-user, no signup, no hosting.
- Runs on the user's own machine, inside Claude Code.
- Must work without writing new integration code beyond a SKILL.md template.
- Must respect the once-a-day, pull-to-read interaction model (no real-time
  popups, no Slack pushes).
- Must be valuable on day one — if `/brief` doesn't earn a daily slot in the
  morning routine, it doesn't matter how clever the architecture is.

## Premises

1. Granola is the only meeting-notes source for v1. Other tools (Otter,
   Read.ai, Fireflies) come later or never.
2. User has a Granola Business or Enterprise plan (or is willing to upgrade) so
   the personal API / MCP server is available. Fallback if not: skill reads
   markdown exports from `~/Documents/granola-exports/`; user manually exports
   each meeting note when it ends. Less magical, still solves the wedge.
3. Calendar source is Google Calendar via `gcalcli` (open source CLI) for v1.
   Apple/Outlook are later additions.
4. The brief is once-a-day, pull-to-read. The user opens Claude Code in the
   morning and types `/brief`. No background process, no scheduled push.
5. Personal tool, single user, runs on the user's own machine.
6. Use Granola's MCP server, not a custom REST integration. Same for any
   calendar source where an MCP option exists.

## Approaches Considered

### Approach A: Cron + CLI script

A standalone bun script run by cron at 7am each morning. Calls Granola MCP
through a small wrapper, calls `gcalcli`, sends the combined context to the
Claude API, writes the result to a markdown file on the desktop.

Pros: survives independent of gstack or Claude Code. Easy to debug.
Cons: no integration with the agent the user is already in; yesterday's briefs
become forgotten markdown files in a folder; doesn't grow into a family of
related commands cleanly.

### Approach B: Local web service

Bun + SQLite running on localhost. Nightly sync from Granola MCP into a local
DB. Web UI on `http://localhost:PORT` with today's brief, person pages, and a
chat box that runs retrieval over the embedded notes. A reminders engine
backed by a cron-driven state machine.

Pros: hits the full v1 vision (brief + person view + chat + reminders).
Browsable history; data compounds. Distributable later if it becomes a
product.
Cons: 4-6 hours of CC+gstack work, ~1-2 weeks of human-equivalent effort. Most
personal-CRM projects get abandoned at the 60% mark; this has the most surface
area to abandon.

### Approach C: `/brief` gstack skill *(recommended)*

A single SKILL.md template installed at `~/.claude/skills/brief/SKILL.md`. When
the user types `/brief` in any Claude Code session, the skill instructs the
agent to:

1. Run `gcalcli agenda --details=description --details=attendees --tsv` for
   today (TSV output is parseable; failure modes: see Error Handling below).
2. For each attendee email, call the Granola MCP server (registered in
   `~/.claude.json` via `claude mcp add granola ...`). Concrete tool call:
   `mcp__granola__search_notes` with query `attendee:<email>`, limit 3, sorted
   by date desc. (Exact tool name to be verified against Granola's MCP schema
   when wiring up — Granola publishes the schema at `granola.ai/mcp/schema`.)
3. Summarize each meeting with: who's there, what was last discussed (one
   sentence), open follow-ups the user committed to (verbatim quote if
   available).
4. Format as a single brief in the terminal output (see Sample Output below).

**Error handling (must-haves for v1):**
- If `gcalcli` is missing or unauthenticated → print install/auth instructions,
  exit cleanly. Do not block on Granola.
- If today has no calendar events → print "No meetings today." and stop.
- If Granola MCP is unreachable → render the calendar-only brief (today's
  meetings + attendees) with a one-line note: "Granola unavailable — past
  context not loaded."
- If an attendee has no Granola history → render their entry with "No prior
  meetings on record" instead of skipping the person.

**Sample output format:**

```
Today, Wednesday May 7

[10:00am] Sarah Chen — 1:1
  Last met 2 weeks ago: discussed hiring plan for Q3.
  You said you'd send: revised JD for the eng manager role.
  Status: not sent yet.

[2:00pm] Kickoff with Acme Corp (3 attendees)
  Marcus Lin (founder, 2 prior meetings):
    Last: pricing pushback on enterprise tier; you offered to revisit.
    Open: send revised pricing deck.
  Priya Rao (eng, 1 prior meeting):
    Last: integration questions about webhook retries.
  No prior history: Tom Walsh.

[4:30pm] Coffee with David — first meeting
  No prior history. Calendar invite says "intro chat re: design feedback."
```

The "tool" is a prompt template. No daemon, no DB, no server. Distribution is
`cp -r brief/ ~/.claude/skills/`. The skill may shell out to `gcalcli`; no
other files needed beyond the SKILL.md itself (the bash blocks live in the
template, gstack-style).

## Recommended Approach

Approach C — `/brief` gstack skill.

Rationale: this is a side project where the goal is to actually use the thing,
not to build the most architecturally complete version. The wedge is "open
Claude Code in the morning, type one command, see today's prep." Approach C
delivers exactly that surface in roughly 15 minutes of CC+gstack time. Future
commands (`/follow-ups`, `/who-said-what`, `/catchup-on-person <name>`) become
sibling skills, not features in a product. If you fall in love with the daily
flow and outgrow the prompt-only architecture, you graduate to Approach B with
a real backend — but that decision is earned by usage, not made up front.

## Open Questions

- **Granola plan tier.** Confirm you're on Business/Enterprise (or check the
  upgrade cost) before relying on the API/MCP path. If you're on free, the
  v1 falls back to "the user manually exports a Granola note as markdown when
  the meeting ends; the skill reads from `~/Documents/granola-exports/`."
- **Calendar identity.** Granola notes are keyed by attendee email (and
  sometimes meeting URL). Google Calendar events have invitee emails. Need to
  confirm Granola exposes attendee email in MCP responses; if it only exposes
  display names, name-matching gets ugly fast for common first names.
- **Time zones.** What "today" means when you're traveling. Probably defer:
  use the local system time zone for v1, revisit if you actually travel.
- **Privacy / sensitivity.** If a Granola note is in a private folder, should
  `/brief` skip it, summarize it, or surface a redacted hint? Default for v1:
  skip private folders entirely.

## Success Criteria

- You type `/brief` in Claude Code in the morning and get a one-screen brief
  fast enough that you don't context-switch away. Rough budget: ~3-8s for
  gcalcli (local, fast) + ~2-4s per attendee for Granola MCP search (5-15
  attendees on a heavy day) + ~5-10s for the agent to compose the brief.
  Total in the 10-30s range; if it stretches past a minute on most days,
  the design has gone wrong and we revisit.
- The brief correctly names today's meetings, today's people, and at least one
  meaningful "last time we talked about" entry per person you've met before.
- You use it ≥3 times in the first week without prompting from anyone.
- At least one follow-up surfaced by the brief (something you committed to in
  a prior meeting) gets actioned the same day. That's the proof that the loop
  closed.

## Distribution Plan

- **Where it lives:** `~/.claude/skills/brief/` on your machine. A single
  SKILL.md inside, no other files.
- **How you install it:** copy the directory in (or symlink it). If you want
  to share it eventually, push it to a public repo and others install with
  `git clone ... ~/.claude/skills/brief`.
- **CI/CD:** none for v1. The skill is a prompt; there's nothing to build.
- **Versioning:** if you graduate to Approach B later, the `/brief` skill stays
  in place and starts calling the local service for richer data. The
  user-facing command name doesn't change.

## Next Steps

1. **Confirm your Granola plan tier.** Check `granola.ai/settings/plan`. If
   you're on Business/Enterprise, request API/MCP access. If you're on free,
   decide whether to upgrade or fall back to manual-export mode.
2. **Install `gcalcli`** and authenticate it with the calendar you want to
   read. (`brew install gcalcli && gcalcli init`.)
3. **Register the Granola MCP server with Claude Code** (one-time, ~30s):
   `claude mcp add granola -- granola mcp` (or whatever invocation Granola's
   docs specify; verify against `granola.ai/mcp` when you set it up).
4. **Hand the design doc to me and say "implement Approach C."** I'll write
   the SKILL.md (bash blocks for gcalcli, prose instructions for the agent on
   how to call Granola MCP and format the brief). No separate helper script.
5. **Use it tomorrow morning.** Type `/brief` before your first meeting. If
   the output isn't useful, we revise the prompt template same day.
6. **After a week of daily use,** decide whether to graduate to Approach B
   (real backend, person pages, chat-over-notes). Don't decide before then.

## Reviewer Concerns (verify at implementation time, not blockers for design)

- **Granola MCP tool name and schema URL are unverified.** Design hedges with
  "tool name to be verified" and a schema URL guessed by convention
  (`granola.ai/mcp/schema`). Before writing the SKILL.md, run `claude mcp list`
  after registering Granola and confirm the actual tool name + parameter shape.
  If the search call doesn't exist, the entire skill no-ops on day one. This
  is the single load-bearing gate.
- **"Private folder" detection is asserted, not specified.** The error-handling
  block says private folders are skipped by default, but the skill needs a
  field on the MCP response to know a note is private. Plan: rely on whatever
  visibility/private flag Granola's MCP returns; if Granola doesn't expose
  visibility in MCP responses, fall back to "skip notes whose folder name
  contains 'private' or 'personal' (case-insensitive)" and revisit.
- **Calendar identity (Open Question 2) is unresolved but Approach C assumes
  email-keyed lookup works.** If Granola's MCP keys notes by display name
  rather than email, common-first-name collisions ("which Sarah?") will surface
  fast. Hedge: when more than one Granola match is returned for an attendee,
  the brief lists both and lets you read context to disambiguate.
