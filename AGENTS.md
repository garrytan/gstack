# gstack-codex development

## Commands

```bash
bun install              # install dependencies
bun test                 # free tests: browse runtime + skill validation
bun run test:smoke       # Codex migration smoke checks
bun run test:exec        # Codex CLI-driven workflow smoke test
bun run dev <cmd>        # run the browse CLI in dev mode
bun run build            # regenerate skills + metadata and compile binaries
bun run gen:skill-docs   # regenerate SKILL.md files from templates
bun run gen:skill-metadata  # regenerate agents/openai.yaml files
bun run skill:check      # skill health dashboard
bun run dev:skill        # watch mode: regen + validate on change
```

`test:exec` assumes the local `codex` CLI is installed and authenticated. The default
test suite stays free and local; the Codex-driven smoke test is opt-in.

## Skill workflow

`SKILL.md` files are generated from `.tmpl` templates. Update the template, run
`bun run gen:skill-docs`, and commit both the template and generated output.

Every shipped skill also has `agents/openai.yaml`, generated by
`bun run gen:skill-metadata`. Treat the template plus metadata generator as the
source of truth, not the generated YAML files.

For intentional divergences from upstream `gstack`, read
`docs/codex-fork-ledger.md` before syncing or rebasing.

## Codex install model

The bundle supports two layouts:

- User install: `~/.codex/skills/gstack-codex`
- Repo-vendored install: `.codex/skills/gstack-codex`

Codex can discover nested skills directly from the bundle root, so we do not rely
on per-skill symlink fanout.

## Writing templates

Template files are prompt instructions for Codex. They are not executable scripts.
Each bash block runs independently.

Rules:
- Use plain English for state and branching logic.
- Prefer bundle-relative file references like `../review/checklist.md`.
- Keep setup instructions valid for both install layouts.
- When a workflow needs a decision, tell Codex to pause and ask in plain text.
- Do not mention Claude-only tools, commands, or path conventions.

## Browser interaction

Use the `browse` skill or the compiled browse binary via `$B <command>`. Do not
fall back to other browser MCP tools for this project.

## Commit style

Keep commits bisectable. Separate mechanical renames, generated file refreshes,
test changes, and behavior changes whenever practical.

## Changelog style

`CHANGELOG.md` is user-facing. Lead with what a user can now do, not the internal
implementation details. Contributor-facing notes belong in a separate contributor
section when they matter.

## Local state

The browser/runtime state stays under `~/.gstack/`. Long-lived local planning docs
can stay under `~/.gstack-dev/` if you need scratch space that should not be committed.
