# 회고 프로토콜

모든 파이프라인 완료 시 자동으로 수행하는 회고(Retrospective) 절차를 정의합니다.
KPT 프레임워크를 사용하며, 정량 데이터 기반으로 개선점을 도출하고,
에이전트 프롬프트와 gotchas에 피드백을 반영하는 루프를 구성합니다.

## 참조 문서

- `references/retro-best-practices.md` -- 회고 진행 시 권장 패턴 및 안티패턴 가이드

## 1. 회고 트리거

회고는 **모든 파이프라인 완료 시 자동 강제**됩니다. 선택 사항이 아닙니다.

| 트리거 조건 | 설명 |
|------------|------|
| 파이프라인 `status: completed` | 정상 완료 — 전체 회고 수행 |
| 파이프라인 `status: failed_at_step_N` | 실패 완료 — 실패 원인 중심 회고 수행 |
| 주간 루틴 (`/bams:weekly`) | 스프린트 종료 — 주간 회고 수행 |

**스킵 불가 원칙:** completion-protocol.md의 Step 5(완료 요약) 출력 전에 회고를 반드시 수행합니다.
사용자가 명시적으로 "회고 건너뛰기"를 요청한 경우에만 `skipped (사용자 선택)` 처리합니다.

## 2. 참여자와 역할

| 참여자 | 역할 |
|--------|------|
| **pipeline-orchestrator** | 회고 진행자 — 프레임워크 진행, 합의 도출, 액션 아이템 확정 |
| **executive-reporter** | 데이터 수집자 — 정량 데이터 수집 및 시각화, 이전 회고 대비 트렌드 제공 |
| **각 부서장** | 부서 대표 — 자기 부서 관점의 KPT 항목 제출, 에이전트 피드백 대리 전달 |

부서장 목록:
- 기획부: product-strategy
- 개발부: frontend-engineering 또는 backend-engineering (해당 파이프라인의 주요 부서장)
- QA부: qa-strategy
- 평가부: product-analytics
- 경영지원: executive-reporter (데이터 수집자 겸임)

## 3. KPT 프레임워크

### 3-1. 수집 단계

각 부서장이 아래 형식으로 KPT 항목을 제출합니다.

#### Keep (유지할 것)
잘 작동한 프로세스, 도구, 패턴, 의사결정을 기록합니다.

```
- [부서명] 내용 (근거: 정량 데이터 또는 구체적 사례)
```

수집 기준:
- 이전 파이프라인 대비 **개선된** 지표가 있는 항목
- 에이전트가 **재사용을 권장**한 패턴
- 사용자가 **만족을 표현**한 결과물

#### Problem (문제점)
장애, 지연, 품질 미달, 재작업이 발생한 항목을 기록합니다.

```
- [부서명] 내용 (영향: 지연 시간, 재작업 횟수, 품질 저하 정도) [일반화 가능성: 다중 프로젝트 재현 가능 / 단일 프로젝트 한정]
```

수집 기준:
- `fail` 상태가 된 Step
- 재시도가 **2회 이상** 발생한 작업
- 에스컬레이션이 발생한 이슈
- 예상 시간 대비 **1.5배 이상** 초과한 Step
- 사용자가 **불만을 표현**한 항목

##### 일반화 가능성 판정 (필수, R1/NF4)

각 Problem에는 다음 중 하나를 반드시 분류 표시한다:

- **다중 프로젝트 재현 가능 (3개 이상)** — 다른 프로젝트/스택에서도 동일 패턴이 재현될 가능성이 높은 일반 행동 규칙·프로세스 이슈. plugin 에이전트 행동 규칙 변경 후보로 분류.
- **단일 프로젝트 한정** — 현재 프로젝트(스택/도메인/외부 서비스/특정 브라우저·OS·디바이스)에 한정된 이슈. plugin 개선이 아닌 `.crew/gotchas.md` 또는 프로젝트 차원 액션 아이템으로 분류한다.

판정 임계값(NF4): **3개 이상 프로젝트에서 재현 가능**한 패턴만 "다중 프로젝트 재현 가능"으로 분류한다. 그 외는 "단일 프로젝트 한정"으로 보수적 분류한다.

표현 원칙(NF6): "특정 브라우저/OS/디바이스 한정 이슈" 같은 일반화된 표현으로 명시한다. 구체 키워드(특정 브라우저/OS 버전 명) 인용은 plugin 외부 컨텍스트에서 의미 보존이 어렵다.

#### Try (시도할 것)
Problem 항목에 대한 구체적 개선 방안을 제안합니다.

```
- [부서명] 내용 (대상 Problem: #N, 예상 효과: ...) [일반화 가능성: 대상 Problem의 분류 상속]
```

수집 기준:
- 모든 Problem에는 **최소 1개**의 Try가 연결되어야 함
- Try는 **다음 파이프라인에서 즉시 적용 가능**한 수준으로 구체화
- "더 잘하겠다" 같은 추상적 표현 금지 — 행동 변경, 설정 변경, 프롬프트 수정 등 구체적 조치
- **일반화 가능성 상속(R2)**: 모든 Try는 대상 Problem의 일반화 가능성 분류를 그대로 상속한다. "단일 프로젝트 한정" Problem에 대한 Try는 plugin 행동 규칙 변경이 아닌 `.crew/gotchas.md` 후보 또는 프로젝트 차원 액션 아이템으로 분류된다.

### 3-2. 합의 단계

pipeline-orchestrator가 수집된 KPT 항목을 종합하고 다음을 결정합니다:

1. **우선순위 정렬**: Problem 항목을 영향도 기준으로 정렬
2. **액션 아이템 확정**: 각 Try에 담당자와 적용 시점을 지정
3. **교차 검증**: 한 부서의 Problem이 다른 부서의 Keep과 모순되지 않는지 확인

## 4. 정량 데이터 수집

executive-reporter가 파이프라인 이벤트 로그에서 수집하는 데이터 목록입니다.

### 4-1. 파이프라인 수준 지표

| 지표 | 소스 | 단위 |
|------|------|------|
| 총 소요 시간 | `pipeline_start` ~ `pipeline_end` 타임스탬프 차이 | 분 |
| Phase별 소요 시간 | 각 Phase의 첫 `step_start` ~ 마지막 `step_end` | 분 |
| Step 성공률 | `done` / 전체 Step 수 | % |
| 재시도 횟수 | 같은 Step에 대한 `step_start` 이벤트 중복 카운트 | 회 |
| 에스컬레이션 횟수 | 에스컬레이션 이벤트 카운트 | 회 |

### 4-2. 에이전트 수준 지표

| 지표 | 소스 | 단위 |
|------|------|------|
| 에이전트별 호출 횟수 | `agent_start` 이벤트 카운트 | 회 |
| 에이전트별 평균 소요 시간 | `agent_start` ~ `agent_end` 평균 | 초 |
| 에이전트별 성공/실패 비율 | `agent_end` 상태 집계 | % |
| 모델별 토큰 사용량 | 에이전트 호출 메타데이터 (가용 시) | 토큰 |

### 4-3. 품질 지표

| 지표 | 소스 | 단위 |
|------|------|------|
| 빌드 성공률 | verify Step 결과 | % |
| 테스트 통과율 | QA Step 결과 | % |
| 린트/타입 에러 수 | verify Step 상세 | 건 |
| 코드 리뷰 지적 사항 수 | review Step 결과 | 건 |
| 보안 이슈 수 | security Step 결과 (실행 시) | 건 |

### 4-4. 트렌드 비교

이전 파이프라인 실행 기록(`.crew/artifacts/pipeline/`)에서 같은 타입의 최근 3회 데이터를 추출하여:
- 현재 실행 vs. 이전 평균 비교
- 개선/악화 방향 표시 (화살표)
- 유의미한 변화(20% 이상)에 주석 추가

## 5. 학습 교훈 저장

회고에서 도출된 교훈은 에이전트 `.md` 파일의 `## 학습된 교훈` 섹션에 기록합니다.

### 5-1. 섹션 형식

에이전트 `.md` 파일 하단에 다음 형식으로 추가합니다:

```markdown
## 학습된 교훈

### [YYYY-MM-DD] 교훈 제목
- **맥락**: 어떤 상황에서 발생했는가
- **문제**: 무엇이 잘못되었거나 비효율적이었는가
- **교훈**: 앞으로 어떻게 해야 하는가
- **적용 범위**: 이 교훈이 적용되는 작업 유형
- **출처**: 파이프라인 slug 또는 회고 ID
```

### 5-2. 기록 규칙

| 규칙 | 설명 |
|------|------|
| 중복 방지 | 같은 맥락+문제의 기존 교훈이 있으면 날짜와 내용을 갱신 (추가하지 않음) |
| 최대 개수 | 에이전트당 최대 10개 유지 |
| 제거 우선순위 | 6개월 이상 된 항목 > 적용 범위가 좁은 항목 > 최근 항목 |
| 사용자 확인 | 에이전트 파일 수정 전 AskUserQuestion으로 확인 |

### 5-3. 교훈 → 행동 연결

기록된 교훈은 해당 에이전트가 다음 호출 시 `## 학습된 교훈` 섹션을 읽고,
현재 작업과 관련된 교훈이 있으면 행동에 반영합니다.

## 6. 개선 피드백 루프

회고 결과가 다음 파이프라인에 반영되는 전체 루프입니다.

```
파이프라인 완료
  │
  ▼
회고 수행 (KPT + 정량 데이터)
  │
  ├─▶ [A] 에이전트 교훈 저장
  │     → 에이전트 .md 파일의 "학습된 교훈" 섹션 업데이트
  │     → 다음 호출 시 에이전트가 교훈 참조
  │
  ├─▶ [B] gotchas 승격 검사
  │     → gotchas-spec.md의 자동 승격 규칙에 따라 검사
  │     → 조건 충족 시 .crew/gotchas.md에 추가 (사용자 확인 후)
  │     → CLAUDE.md Gotchas 동기화 (completion-protocol.md Step 3 참조)
  │
  ├─▶ [C] Pipeline Learnings 업데이트
  │     → .crew/config.md의 Pipeline Learnings 섹션 갱신
  │     → pipeline-learnings-taxonomy.md 형식 준수
  │
  └─▶ [D] 프로세스 개선
        → 반복 발생 Problem은 references/ 프로토콜 수정 제안
        → 에이전트 프롬프트(행동 규칙) 수정 제안
        → 사용자 승인 후 반영
```

### 6-1. 피드백 적용 우선순위

| 우선순위 | 적용 대상 | 적용 방식 | 승인 |
|----------|----------|----------|------|
| 1 (즉시) | Pipeline Learnings | config.md 자동 갱신 | 불필요 |
| 2 (즉시) | 에이전트 교훈 | 에이전트 .md 파일 갱신 | AskUserQuestion |
| 3 (조건부) | gotchas | gotchas.md + CLAUDE.md | AskUserQuestion |
| 4 (제안) | 프로토콜/프롬프트 수정 | 수정안 제시 후 사용자 판단 | 명시적 승인 필요 |

### 6-2. 회고 결과 저장

회고 결과는 파이프라인 tracking 파일에 다음 형식으로 기록합니다:

```yaml
retro:
  conducted_at: [ISO timestamp]
  keep_count: N
  problem_count: N
  try_count: N
  action_items:
    - description: "..."
      assignee: [에이전트명]
      target: "다음 파이프라인"
      status: pending
  lessons_saved: [에이전트 목록]
  gotchas_promoted: [승격된 gotchas 수]
  learnings_updated: [갱신된 learnings 수]
```

### 6-3. 다음 파이프라인에서의 참조

다음 파이프라인의 Pre-flight(preflight-protocol.md)에서:
1. 이전 회고의 `action_items` 중 `status: pending`인 항목을 확인
2. 해당 항목의 담당 에이전트에게 "이전 회고에서 합의된 개선 사항"으로 전달
3. 파이프라인 완료 시 해당 액션 아이템의 적용 여부를 확인하고 상태 갱신
