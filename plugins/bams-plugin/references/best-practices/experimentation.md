# Experimentation Best Practices

## Responsibility: 실험 설계, 실험 계측 검증, 결과 해석, 표본 크기 산정, 편향 탐지

---

### 실험 설계 (design_experiment)

**언제 참조:** ux-designer 또는 product-analytics로부터 A/B 테스트 설계 요청을 받았을 때

**협업 대상:**
- product-analytics: 주 지표 및 가드레일 지표 합의
- ux-designer: 실험군/대조군 UI 정의 수신
- frontend-engineering / backend-engineering: 피처 플래그 구현 협업

**작업 절차:**
1. 가설을 명확한 문장으로 정의한다: "[변경 사항]을 하면 [지표]가 [방향]으로 변화할 것이다"
2. 주 지표(primary metric)를 1개로 한정하고 가드레일 지표를 정의한다
3. 표본 크기와 통계적 검정력(power ≥ 80%, α = 0.05)을 계산한다
4. 실험 종료 조건(표본 크기 도달 또는 최대 기간)을 사전에 정의한다
5. 실험 코드(피처 플래그) 구현 스펙을 frontend-engineering에게 전달한다

**산출물:** 실험 설계서 (가설, 주/가드레일 지표, 표본 크기, 종료 조건, 피처 플래그 스펙)

**주의사항:**
- 주 지표는 반드시 1개다 — 여러 지표를 동시에 최적화하면 통계적 유의성이 희석된다
- 실험 시작 후 종료 조건을 임의로 변경하지 않는다 — Peeking Problem(조기 종료 편향)이 발생한다

### 실험 계측 검증 (verify_experiment_instrumentation)

**언제 참조:** 실험 시작 전 이벤트 수집이 올바르게 설정되었는지 검증할 때

**협업 대상:**
- data-integration: 이벤트 트래킹 구현 확인
- product-analytics: 이벤트가 분석 가능한 구조로 수집되는지 확인

**작업 절차:**
1. 실험 그룹 배정이 올바르게 작동하는지 확인한다 (A/B 비율 검증)
2. 주 지표와 가드레일 지표의 이벤트 수집 여부를 검증한다
3. 테스트 환경에서 이벤트 페이로드가 정의된 스키마와 일치하는지 확인한다
4. 실험 시작 후 24시간 내 데이터가 정상 수집되는지 모니터링한다

**산출물:** 계측 검증 보고서 (그룹 배정 확인, 이벤트 수집 상태, 데이터 샘플)

**주의사항:**
- 계측 검증 없이 실험을 시작하면 결과 해석 불가로 실험 전체가 무효화된다
- 이벤트 수집 누락이 있으면 실험을 즉시 중단하고 data-integration에게 수정을 요청한다

### 결과 해석 (interpret_results)

**언제 참조:** 실험이 종료된 후 결과를 분석하고 의사결정을 도출할 때

**협업 대상:**
- product-analytics: 통계적 유의성 검증 협업
- product-strategy: 결과 기반 제품 방향 결정 지원
- business-kpi: 실험 결과가 사업 목표에 미치는 영향 분석

**작업 절차:**
1. 통계적 유의성을 검증한다 (p < 0.05, 신뢰구간 확인)
2. 가드레일 지표에 부정적 영향이 있는지 확인한다
3. 실용적 유의성(Effect Size)을 확인한다 — 통계적으로 유의해도 효과 크기가 작으면 실제 가치가 없다
4. 결과를 다음 3가지로 분류한다: 승리(B 채택), 패배(A 유지), 결론 없음(추가 실험 필요)
5. 다음 행동을 권고한다: 전체 배포, 재실험, 가설 수정 등

**산출물:** 실험 결과 보고서 (통계적 유의성, 효과 크기, 가드레일 영향, 권고 사항)

**주의사항:**
- p-value가 0.05 미만이어도 효과 크기가 무시할 수준이면 채택을 보류한다
- "결론 없음" 결과도 유효한 결과다 — 가설이 틀렸다는 것을 알게 된 것도 학습이다

### 편향 탐지 (detect_bias)

**언제 참조:** 실험 결과가 예상과 다를 때, 또는 실험 설계의 유효성을 검토할 때

**협업 대상:**
- data-integration: 이벤트 수집 편향 확인
- product-analytics: 데이터 이상 여부 교차 확인

**작업 절차:**
1. 선택 편향(Selection Bias): A/B 그룹의 사용자 특성이 동등한지 검증한다 (SRM 검사)
2. 노출 편향: 모든 실험군 사용자가 실제로 변경 사항에 노출되었는지 확인한다
3. 노벨티 효과: 초기 반응이 시간이 지남에 따라 정상화되는지 추적한다
4. 네트워크 효과: A/B 그룹이 서로 영향을 미치는지 확인한다 (특히 소셜 기능에서 발생)

**산출물:** 편향 탐지 보고서 (탐지된 편향 목록, 영향 평가, 보정 방법)

**주의사항:**
- SRM(Sample Ratio Mismatch)이 감지되면 실험을 즉시 무효화한다 — 결과를 신뢰할 수 없다
- 노벨티 효과가 의심될 때는 실험 기간을 연장하여 초기 반응이 안정화될 때까지 관찰한다
