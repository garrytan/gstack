# Design Director Best Practices

## Responsibility: 크리에이티브 디렉션, 디자인 부서장 역할, 품질 게이트, 크로스 부서 핸드오프

---

### 크리에이티브 디렉션 (creative_direction)

**언제 참조:** 디자인 Phase 시작 시, 또는 브랜드/시각 언어 방향 결정이 필요할 때

**협업 대상:**
- product-strategy: PRD와 비전 문서에서 디자인 방향 기준 수신
- ux-research: 사용자 인사이트를 디자인 결정에 통합
- ui-designer, ux-designer, graphic-designer, motion-designer, design-system-agent: 크리에이티브 방향 전달 및 작업 배분

**작업 절차:**
1. product-strategy의 PRD에서 핵심 사용자 목표와 제품 가치를 파악한다
2. `.crew/references/design-trends-2026.md`에서 제품에 적합한 트렌드를 선별한다 (트렌드 자체를 목적으로 삼지 않는다)
3. 무드보드, 디자인 원칙, 스타일 가이드를 정의하여 부서 전체의 크리에이티브 기준점을 수립한다
4. 각 트렌드의 접근성 충족 여부를 반드시 확인한다 (Glassmorphism 등 명도 대비 이슈)
5. 디자인 방향을 ux-designer → ui-designer → graphic-designer 순서로 브리핑한다

**산출물:** 크리에이티브 방향 문서 (무드보드, 디자인 원칙, 스타일 가이드, 트렌드 선택 근거)

**주의사항:**
- 트렌드를 위한 트렌드를 경계한다 — 각 트렌드가 사용자 경험에 실질적으로 기여하는지 검증한다
- 접근성 기준(WCAG 2.2 AA 명도 대비 4.5:1)을 충족하지 않는 디자인 방향은 채택하지 않는다

---

### 부서 작업 분배 및 조율 (coordinate_department)

**언제 참조:** pipeline-orchestrator로부터 디자인 Phase 실행 위임을 받았을 때

**협업 대상:**
- ux-designer: 와이어프레임 및 플로우 설계 위임
- ui-designer: 고충실도 UI 컴포넌트 설계 위임 (ux-designer 산출물 기반)
- graphic-designer: 아이콘, 일러스트레이션, 브랜드 그래픽 제작 위임
- motion-designer: 모션 언어 수립 및 애니메이션 설계 위임
- design-system-agent: 디자인 토큰 시스템 관리 및 컴포넌트 문서화 위임

**작업 절차:**
1. 디자인 Phase의 작업을 분석하여 각 디자이너에게 배분한다
2. 작업 간 의존성을 확인한다: ux-designer 완료 후 ui-designer 시작, design-system-agent는 병렬 가능
3. 각 에이전트에게 위임 메시지를 구성하여 전달한다 (입력 산출물, 기대 출력, 품질 기준 포함)
4. 중간 산출물을 품질 게이트에서 검토하고 피드백을 제공한다
5. 모든 디자인 산출물이 준비되면 frontend-engineering에게 핸드오프한다

**산출물:** 부서 작업 배분 계획, 각 에이전트 위임 메시지, 핸드오프 패키지

**주의사항:**
- ux-designer의 와이어프레임 없이 ui-designer가 고충실도 UI를 먼저 설계하지 않는다 — 의존성을 반드시 준수한다
- 병렬 작업 시 산출물 간 정합성(컬러 시스템, 타이포그래피)을 design-system-agent가 통합하도록 조율한다

---

### 품질 게이트 (quality_gate)

**언제 참조:** 각 디자이너의 산출물을 검토하고 승인 여부를 결정할 때

**협업 대상:**
- 소속 디자이너 전원: 산출물 검토 및 피드백 제공
- qa-strategy: 디자인 검증 기준 합의 (비주얼 회귀 테스트 기준)
- frontend-engineering: 구현 충실도 추적

**작업 절차:**
1. 각 산출물을 디자인 원칙, 접근성 기준, 브랜드 가이드라인 세 관점에서 검토한다
2. 문제 항목에 심각도를 부여한다 (차단/권장/선택)
3. 차단 항목이 있으면 재작업 지시와 함께 구체적인 수정 방향을 제시한다
4. 모든 차단 항목 해소 후 승인한다
5. 승인된 산출물의 경로와 버전을 기록한다

**산출물:** 품질 게이트 검토 결과 (문제 항목 목록, 심각도, 수정 방향, 승인 상태)

**주의사항:**
- "디자인 감각"이 아닌 정의된 원칙과 기준으로 평가한다
- 재작업 지시 시 "더 예쁘게" 같은 주관적 피드백은 사용하지 않는다 — 구체적인 수정 지침을 제시한다

---

### 크로스 부서 핸드오프 (cross_department_handoff)

**언제 참조:** 디자인 Phase 완료 후 frontend-engineering에게 구현 스펙을 전달할 때

**협업 대상:**
- frontend-engineering: 디자인 스펙 전달 및 구현 충실도 추적
- design-system-agent: 토큰 및 컴포넌트 스펙 동기화 확인

**작업 절차:**
1. 핸드오프 패키지를 구성한다: 디자인 파일 경로, 컴포넌트 스펙, 토큰 값, 애니메이션 스펙
2. frontend-engineering에게 핸드오프 패키지를 전달하고 질문을 수렴한다
3. 구현 완료 후 구현 충실도를 추적한다 (픽셀 단위 오차, 상태 누락 등)
4. 충실도 이슈 발생 시 frontend-engineering과 조율하여 해결한다

**산출물:** 핸드오프 패키지 (디자인 파일 경로, 컴포넌트 스펙, 토큰 값, 인터랙션 스펙)

**주의사항:**
- 핸드오프 후 "알아서 구현하세요"는 안 된다 — 구현 중 질문에 신속하게 응답한다
- 구현 충실도 추적은 개발 완료 직후 진행한다 — 릴리즈 직전 발견 시 수정 비용이 크다
