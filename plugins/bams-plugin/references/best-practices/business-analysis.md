# Business Analysis Best Practices

## Responsibility: 요구사항 도출, 기능 명세 작성, 스코프 분해

---

### 요구사항 도출 (elicit_requirements)

**언제 참조:** product-strategy로부터 PRD 초안을 받아 상세 기능 명세를 도출할 때

**협업 대상:**
- product-strategy: 제품 비전과 우선순위 기준 확인, PRD 초안 수신
- ux-research: 사용자 관점 요구사항 검증
- frontend-engineering: 프론트엔드 구현 제약 사전 확인
- backend-engineering: 백엔드 구현 제약 및 API 스키마 협의

**작업 절차:**
1. product-strategy의 PRD 초안을 분석하여 암묵적 요구사항을 표면화한다
2. 모호한 표현(빠르게, 편하게, 잘 등)을 발견하면 정량 기준으로 치환한다
3. "~하면 좋겠다(should)"와 "~해야 한다(must)"를 분리하여 우선순위를 반영한다
4. 상충하는 요구사항을 식별하고 product-strategy에게 조정을 요청한다
5. 기존 코드베이스를 Glob, Read로 확인하여 현재 구현과의 정합성을 검증한다

**산출물:** 요구사항 명세서 (이해관계자 목록, 기능/비기능 요구사항, 업무 규칙, 범위 외 항목, 미결 질문)

**주의사항:**
- 요구사항마다 출처(누가 요청했는가)와 목적(왜 필요한가)을 반드시 기록한다
- 상충 요구사항을 방치하면 구현 중 재작업이 발생한다 — 즉시 해소한다

---

### 기능 명세 작성 (write_functional_spec)

**언제 참조:** 요구사항이 확정된 후 개발팀이 바로 구현할 수 있는 수준의 명세를 작성할 때

**협업 대상:**
- frontend-engineering: 프론트엔드 기능 명세 전달 — 구현 시 추가 질문 없이 진행 가능한 수준
- backend-engineering: 백엔드 기능 명세 및 API 스키마 전달
- qa-strategy: 인수 기준을 테스트 케이스로 연계

**작업 절차:**
1. 정상 흐름(Happy Path)을 먼저 정의한 후 예외 흐름(Edge Case)을 추가한다
2. 각 기능에 사전 조건, 사후 조건, 불변 조건을 명시한다
3. 입력 데이터의 유효성 규칙(타입, 범위, 필수 여부)을 빠짐없이 기술한다
4. 상태 전이가 있는 기능은 허용/금지 전이를 명시적으로 나열한다
5. 개발자가 추가 질문 없이 구현할 수 있는 수준을 목표로 작성한다

**산출물:** 기능 명세서 (개요, 사전/사후 조건, 정상 흐름, 예외 흐름, 입력 데이터 규칙, 상태 전이, 영향 범위)

**주의사항:**
- "개발자가 판단하겠지"라는 생각은 버린다 — 모든 경계 조건을 명시한다
- 비기능 요구사항(성능, 보안, 접근성)을 기능 명세와 함께 기술한다

### 스코프 분해 (decompose_scope)

**언제 참조:** 대규모 기능 요구사항을 구현 가능한 단위로 분해할 때

**협업 대상:**
- project-governance: 분해 결과의 일정 반영
- pipeline-orchestrator: 분해된 태스크를 Phase/Step으로 매핑

**작업 절차:**
1. 대규모 요구사항을 독립적으로 구현, 테스트, 배포 가능한 단위로 분할한다 (INVEST 원칙)
2. 각 작업 단위가 1~3일 내 완료 가능한 크기인지 검증한다
3. 작업 간 의존성을 명시하고 순환 의존을 제거한다
4. 분해 결과가 원래 요구사항을 완전히 커버하는지 역추적 매핑으로 검증한다

**산출물:** 스코프 분해 문서 (작업 목록, 의존성 다이어그램, 역추적 매핑)

**주의사항:**
- INVEST 원칙: Independent(독립적), Negotiable(협의 가능), Valuable(가치 있는), Estimable(추정 가능), Small(작은), Testable(테스트 가능)
- 분해 후 역추적 매핑 없이 product-strategy에게 보고하지 않는다

### 업무 규칙 정형화 (formalize_business_rules)

**언제 참조:** 복잡한 비즈니스 로직(조건 분기, 상태 전이, 권한 규칙)을 명세할 때

**협업 대상:**
- backend-engineering: 정형화된 규칙을 서버 로직 구현에 직접 활용
- qa-strategy: 결정 테이블을 테스트 케이스 설계에 활용

**작업 절차:**
1. 조건-행동 쌍을 결정 테이블(Decision Table)로 정리한다
2. 누락된 조건 조합이 있는지 빠짐없이 검토한다
3. 예외 조건과 에러 처리 정책을 정상 흐름과 동일한 수준으로 명세한다
4. 상태 전이 다이어그램을 작성하여 허용/금지 전이를 시각화한다

**산출물:** 업무 규칙 정형화 문서 (결정 테이블, 상태 전이 다이어그램, 에러 처리 정책)

**주의사항:**
- "나머지 경우" 처리를 반드시 명시한다 — 누락된 조건 조합이 런타임 에러의 원인이 된다
- 결정 테이블의 각 행은 서로 배타적(mutually exclusive)이어야 한다
