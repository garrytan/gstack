# Performance Evaluation Best Practices

## Responsibility: 성능 벤치마크 정의, 부하 및 안정성 검증, 체감 성능 분석, 병목 진단, 성능 회귀 탐지

---

### 성능 벤치마크 정의 (define_performance_benchmark)

**언제 참조:** 새 피처 구현 전 성능 기준을 설정할 때, 또는 릴리즈 전 성능 게이트를 정의할 때

**협업 대상:**
- platform-devops: 성능 측정 인프라 및 모니터링 설정 협업
- product-analytics: Core Web Vitals와 사용자 이탈률 간 상관관계 분석
- qa-strategy: 성능 기준을 QA 체크리스트에 포함

**작업 절차:**
1. 응답 시간(p50, p95, p99), 처리량(RPS), 오류율 등 합격 기준을 수치로 정의한다
2. 사용자 체감 지표(Core Web Vitals)와 백엔드 지표를 함께 정의한다
   - LCP(Largest Contentful Paint) < 2.5s
   - FID(First Input Delay) < 100ms
   - CLS(Cumulative Layout Shift) < 0.1
3. 테스트 환경과 프로덕션 환경의 차이(하드웨어, 네트워크)를 문서화한다
4. 기준 미달 시 릴리즈 차단 조건으로 설정할 지표를 release-quality-gate와 합의한다

**산출물:** 성능 벤치마크 정의서 (지표 목록, 합격 기준, 측정 방법, 환경 명세)

**주의사항:**
- p95 대신 p50만 기준으로 삼으면 꼬리 지연(tail latency)을 놓친다 — 반드시 p95/p99를 포함한다
- 개발 환경의 성능은 프로덕션보다 좋거나 나쁠 수 있다 — 환경 차이를 명시한다

### 부하 및 안정성 검증 (verify_load_stability)

**언제 참조:** 릴리즈 전 피크 트래픽 또는 대규모 부하에서의 안정성을 검증할 때

**협업 대상:**
- automation-qa: 부하 테스트 스크립트 공동 작성
- platform-devops: 부하 테스트 인프라 구성 및 모니터링
- backend-engineering: 병목 지점 발견 시 최적화 협업

**작업 절차:**
1. 예상 피크 트래픽(RPS)을 정의하고 테스트 시나리오를 설계한다 (점진적 부하 증가, 급격한 스파이크)
2. 단일 측정이 아닌 반복 측정의 통계값(평균, 표준편차)을 기준으로 판단한다
3. 부하 테스트 중 CPU, 메모리, DB 연결 수, 네트워크 사용률을 모니터링한다
4. 성능 저하 임계점(Degradation Point)을 식별한다
5. 결과를 재현 가능한 조건과 함께 보고한다

**산출물:** 부하 테스트 결과 보고서 (시나리오, 부하 단계별 성능 수치, 병목 지점, 권고 사항)

**주의사항:**
- 단일 측정으로 결론을 내리지 않는다 — 최소 3회 반복 측정 후 통계값을 사용한다
- 부하 테스트는 반드시 프로덕션과 격리된 환경에서 실행한다 (프로덕션 트래픽에 영향 없이)

### 병목 진단 (diagnose_bottleneck)

**언제 참조:** 성능 기준 미달의 원인을 파악할 때, 또는 성능 저하 증상이 보고될 때

**협업 대상:**
- backend-engineering: DB 쿼리 최적화, 캐시 전략, 비동기 처리 개선 협업
- frontend-engineering: 렌더링 병목, 번들 크기 최적화 협업
- platform-devops: 인프라 레벨 병목(네트워크, CPU/메모리 제한) 확인

**작업 절차:**
1. 병목을 계층별로 식별한다: 프론트엔드(렌더링), 네트워크, API, 데이터베이스, 인프라
2. 프로파일링 도구를 사용하여 가장 느린 구간을 정량화한다
3. 이슈를 재현 가능한 조건(요청 수, 데이터 크기, 동시 접속 수)으로 정의한다
4. 개선 방안을 제안하고 개선 전후 수치 비교를 통해 효과를 검증한다

**산출물:** 병목 진단 보고서 (병목 위치, 재현 조건, 수치 증거, 개선 방안, 전후 비교)

**주의사항:**
- 추측으로 최적화하지 않는다 — 측정으로 병목을 확인한 후 최적화한다
- 하나의 병목을 해결하면 다른 병목이 드러날 수 있다 — 해결 후 재측정한다

### 성능 회귀 탐지 (detect_performance_regression)

**언제 참조:** 릴리즈 간 성능 변화를 추적하여 회귀를 조기 발견할 때

**협업 대상:**
- platform-devops: CI/CD 파이프라인에 성능 측정 통합
- product-analytics: 성능 지표와 사용자 행동(이탈률) 상관관계 분석

**작업 절차:**
1. 베이스라인(기준 릴리즈)의 성능 수치를 기록한다
2. 각 릴리즈에서 베이스라인 대비 성능 변화를 측정한다
3. 통계적 유의성 있는 성능 저하(>10%)가 감지되면 릴리즈를 차단한다
4. 성능 회귀의 원인이 된 커밋을 이분 탐색(git bisect)으로 식별한다

**산출물:** 성능 회귀 탐지 보고서 (릴리즈 간 성능 비교 차트, 회귀 커밋 식별, 영향 범위)

**주의사항:**
- 베이스라인 없이는 회귀를 탐지할 수 없다 — 첫 측정부터 베이스라인을 기록한다
- 성능 회귀가 발견되면 원인 커밋을 반드시 식별한다 — "성능이 나빠졌음" 보고만으로는 수정이 불가능하다
