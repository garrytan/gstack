# Backend Engineering Best Practices

## Responsibility: API 계약 설계, 도메인 로직 구현, 데이터 정합성 관리

---

### API 계약 설계 (design_api_contract)

**언제 참조:** 새 기능의 백엔드 API를 설계할 때, 또는 프론트엔드와 인터페이스를 합의할 때

**협업 대상:**
- frontend-engineering: API 인터페이스 합의 — 엔드포인트, 요청/응답 스키마, 에러 코드 체계
- data-integration: 이벤트 스키마 합의 — 어떤 행동 이벤트를 발행할지 사전 합의
- business-analysis: 기능 명세에서 API 요구사항 도출

**작업 절차:**
1. 기존 API 구조를 Glob, Read로 파악하여 프로젝트 컨벤션(RESTful, GraphQL 등)을 따른다
2. 엔드포인트 경로, HTTP 메서드, 요청/응답 스키마를 정의한다
3. 에러 코드 체계를 명확히 정의한다 (4xx vs. 5xx, 에러 코드 의미)
4. 페이지네이션, 정렬, 필터링 규칙을 명세한다
5. 버전 관리 전략(URL 버전 vs. 헤더 버전)을 결정한다
6. API 계약을 frontend-engineering에게 전달하기 전에 business-analysis의 기능 명세와 정합성을 검증한다

**산출물:** API 계약 문서 (엔드포인트 목록, 요청/응답 스키마, 에러 코드 체계, 페이지네이션 규칙)

**주의사항:**
- API 계약을 먼저 확정하고 구현을 시작한다 — frontend와 backend가 계약 없이 병렬 작업하면 나중에 인터페이스 충돌이 발생한다
- 하위 호환성 없는 API 변경은 버전 업그레이드로 처리한다

### 도메인 로직 구현 (implement_domain_logic)

**언제 참조:** business-analysis의 업무 규칙을 서버 로직으로 구현할 때

**협업 대상:**
- business-analysis: 업무 규칙 및 상태 전이 명세 수신
- data-integration: 도메인 이벤트 발행 — 상태 전이 시 이벤트를 발행하여 다른 시스템과 느슨하게 결합
- qa-strategy: 도메인 로직 인수 기준 확인

**작업 절차:**
1. business-analysis의 결정 테이블과 상태 전이 다이어그램을 확인한다
2. 비즈니스 불변식(Invariant)을 코드로 표현한다
3. 상태 전이 규칙을 명시적으로 관리한다 (허용 전이, 금지 전이)
4. 도메인 이벤트를 발행하여 시스템 간 느슨한 결합을 유지한다
5. 기존 코드베이스의 패턴을 Read로 확인하여 일관성을 유지한다

**산출물:** 도메인 로직 구현 코드, 도메인 이벤트 정의, 단위 테스트

**주의사항:**
- 비즈니스 로직을 컨트롤러/라우터에 직접 작성하지 않는다 — 서비스 레이어로 분리한다
- 도메인 이벤트 발행 실패 시 메인 트랜잭션이 롤백되지 않도록 비동기 처리한다

### 데이터 정합성 관리 (manage_data_integrity)

**언제 참조:** 데이터 스키마 변경, 마이그레이션 계획 수립, 또는 동시성 이슈 해결 시

**협업 대상:**
- data-integration: 데이터 스키마 변경 시 이벤트 스키마 영향 분석
- platform-devops: 마이그레이션 배포 절차 협의 — 다운타임 없는 롤링 마이그레이션
- qa-strategy: 데이터 정합성 테스트 케이스 협의

**작업 절차:**
1. 스키마 변경 전 영향 범위(기존 데이터, 쿼리, API)를 분석한다
2. 마이그레이션을 하위 호환성 유지 방식으로 설계한다 (Add-only, Expand-then-Contract)
3. 트랜잭션 경계를 명확히 설정하고 ACID 속성을 확인한다
4. 동시성 이슈(Race Condition, Deadlock)를 방지하기 위한 Optimistic/Pessimistic Locking을 선택한다
5. 인덱스 전략을 검토하여 쿼리 성능을 확보한다

**산출물:** 마이그레이션 스크립트, 트랜잭션 설계, 인덱스 계획

**주의사항:**
- 마이그레이션은 항상 롤백 가능하게 설계한다 (Up/Down 쌍으로 작성)
- 대용량 테이블 마이그레이션은 배치 처리로 분할하여 잠금 최소화한다
- 스키마 변경 후 data-integration의 이벤트 스키마 영향을 반드시 확인한다
