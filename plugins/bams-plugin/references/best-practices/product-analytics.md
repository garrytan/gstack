# Product Analytics Best Practices

## Responsibility: KPI 정의 및 표준화, 퍼널 및 행동 분석, 릴리즈 영향 평가, 코호트 분석, 이상 탐지, 평가부서 부서장

---

### KPI 정의 및 표준화 (define_kpi)

**언제 참조:** 새 피처의 성공 기준을 정의할 때, 또는 평가 Phase 시작 시

**협업 대상:**
- business-kpi: KPI 정의를 사업 목표와 정렬
- data-integration: 정의된 KPI를 수집할 이벤트 트래킹 요구사항으로 전환
- product-strategy: KPI가 제품 비전의 성공 지표와 일치하는지 검토

**작업 절차:**
1. North Star Metric(단일 핵심 지표)을 정의하고 Product KPI와의 연결 관계를 명시한다
2. 각 KPI의 산출 공식, 측정 주기, 데이터 소스를 정의한다
3. KPI 측정에 필요한 이벤트 트래킹을 data-integration에게 요청한다
4. 이상 탐지 알림 임계값을 설정한다

**산출물:** KPI 정의서 (North Star Metric, Product KPI, 산출 공식, 데이터 소스, 알림 임계값)

**주의사항:**
- KPI가 너무 많으면 집중도가 떨어진다 — North Star Metric 1개, 보조 지표 최대 5개로 제한한다
- 측정 불가능한 KPI는 KPI가 아니다 — 데이터 소스가 확보된 지표만 정의한다

### 퍼널 및 행동 분석 (analyze_funnel_behavior)

**언제 참조:** 기능 사용 행태를 분석하거나 전환율 저하 원인을 파악할 때

**협업 대상:**
- data-integration: 이벤트 데이터 수집 상태 확인 및 누락 이벤트 추가 요청
- ux-research: 정량 분석 결과와 정성 리서치 교차 검증
- experimentation: 가설 도출 후 A/B 테스트 설계 협업

**작업 절차:**
1. 분석 대상 퍼널을 정의한다 (예: 랜딩 → 가입 → 첫 사용 → 재방문)
2. 각 단계의 전환율을 계산하고 이탈이 가장 큰 지점을 식별한다
3. 이탈 지점에서의 사용자 행동 패턴을 분석한다
4. 이탈 원인 가설을 도출하고 experimentation에게 A/B 테스트를 요청한다

**산출물:** 퍼널 분석 보고서 (단계별 전환율, 주요 이탈 지점, 행동 패턴, 개선 가설)

**주의사항:**
- 상관관계를 인과관계로 해석하지 않는다 — 가설은 실험으로 검증한다
- 세그먼트(신규/재방문, 디바이스 유형)별로 분석하여 숨겨진 패턴을 발견한다

### 릴리즈 영향 평가 (evaluate_release_impact)

**언제 참조:** 피처 릴리즈 후 KPI에 미친 영향을 평가할 때

**협업 대상:**
- business-kpi: 릴리즈가 사업 지표에 미친 영향 평가
- executive-reporter: 릴리즈 영향 평가 결과를 경영진 보고서에 포함
- product-strategy: 릴리즈 결과를 다음 로드맵 우선순위 결정에 반영

**작업 절차:**
1. 릴리즈 전후 KPI를 비교하는 기간을 정의한다 (최소 2주 전후 비교 권장)
2. 계절성, 캠페인 등 외부 요인의 영향을 보정한다
3. 통계적 유의성을 확인한다 (신뢰구간, p-value)
4. 릴리즈 영향이 긍정적이면 확장 방안을, 부정적이면 수정 방안을 제안한다

**산출물:** 릴리즈 영향 평가 보고서 (전후 KPI 비교, 통계적 유의성, 외부 요인 분석, 권고 사항)

**주의사항:**
- 2주 미만의 데이터로 릴리즈 영향을 확정적으로 판단하지 않는다 — 초기 트렌드임을 명시한다
- 외부 요인(명절, 마케팅 캠페인)을 고려하지 않으면 잘못된 결론을 도출한다

### 이상 탐지 (detect_anomaly)

**언제 참조:** 지표의 급격한 변화나 비정상 패턴이 감지될 때

**협업 대상:**
- business-kpi: 사업 지표 이상 발생 시 즉시 통보
- platform-devops: 기술적 원인(서비스 장애, 배포 이슈) 확인 협업
- executive-reporter: 이상 탐지 결과를 경영진에게 보고

**작업 절차:**
1. 이상 탐지 알림을 수신하면 먼저 데이터 수집 파이프라인 문제인지 확인한다 (실제 이상 vs. 데이터 이상)
2. 이상이 시작된 시점을 정확히 파악하고 해당 시점의 이벤트(배포, 캠페인 등)와 연관 짓는다
3. 영향 범위(전체 사용자 vs. 특정 세그먼트)를 분석한다
4. 원인 가설을 최소 2개 제시하고 각 가설 검증 방법을 명시한다

**산출물:** 이상 탐지 보고서 (이상 시작 시점, 영향 범위, 원인 가설, 검증 방법)

**주의사항:**
- 데이터 수집 파이프라인 문제를 실제 제품 이슈로 혼동하지 않는다 — data-integration과 먼저 확인한다
- 이상 탐지 알림이 너무 자주 울리면 alert fatigue가 발생한다 — 임계값을 주기적으로 재검토한다
