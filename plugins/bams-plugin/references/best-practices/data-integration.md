# Data Integration Best Practices

## Responsibility: 이벤트 트래킹 구현, 외부 시스템 연동, 데이터 파이프라인 관리, 데이터 스키마 관리

---

### 이벤트 트래킹 구현 (implement_event_tracking)

**언제 참조:** 새 피처 구현 시 행동 데이터 수집을 추가할 때, 또는 이벤트 스키마를 변경할 때

**협업 대상:**
- backend-engineering: 이벤트 스키마 합의 — 서버 이벤트와 클라이언트 이벤트의 경계 정의
- product-analytics: 추적 요구사항 수신 — 어떤 이벤트를 수집해야 분석이 가능한지 확인
- frontend-engineering: 클라이언트 이벤트 발행 위치 및 페이로드 합의

**작업 절차:**
1. product-analytics에서 추적 요구사항을 수신한다
2. 이벤트 이름은 `[대상]_[행위]` 형식으로 일관되게 명명한다 (예: `product_viewed`, `payment_completed`)
3. 모든 이벤트에 공통 필드(timestamp, user_id, session_id, event_id)를 포함한다
4. 이벤트 스키마를 버전 관리하고 하위 호환성을 유지한다
5. 이벤트 페이로드에 PII(개인식별정보)를 포함하지 않거나 마스킹 처리한다
6. 이벤트 발행 실패 시 로컬 큐에 저장하고 재발행을 시도하는 안전장치를 구현한다

**산출물:** 이벤트 트래킹 구현 코드, 이벤트 스키마 정의, PII 마스킹 로직

**주의사항:**
- 이벤트 중복 발행을 방지하기 위해 event_id로 멱등성을 확보한다
- 이벤트 발행 실패가 메인 기능에 영향을 미치지 않도록 비동기 처리한다

### 외부 시스템 연동 (integrate_external_systems)

**언제 참조:** 결제, 인증, 메시지, 파트너 시스템과의 연동을 구현할 때

**협업 대상:**
- backend-engineering: 외부 API 어댑터를 백엔드 도메인 로직과 통합
- platform-devops: 외부 시스템 연동 실패 시 알림 및 모니터링 설정

**작업 절차:**
1. 외부 API의 계약(인증 방식, 요청/응답 형식, 에러 코드)을 확인한다
2. 어댑터 패턴을 사용하여 외부 시스템 변경에 유연하게 대응한다
3. 타임아웃, 재시도(Exponential Backoff), 서킷 브레이커 패턴을 적용한다
4. 연동 테스트를 위한 Mock/Stub을 구현하여 독립적인 테스트가 가능하게 한다
5. 외부 시스템의 SLA와 내부 SLA의 차이를 완충하는 전략을 포함한다

**산출물:** 외부 시스템 어댑터 코드, 재시도/서킷 브레이커 설정, Mock 구현

**주의사항:**
- 외부 API 키와 시크릿은 환경 변수로 관리한다 — 코드에 직접 포함 절대 금지
- 서킷 브레이커가 Open 상태일 때 사용자에게 적절한 fallback을 제공한다

### 데이터 파이프라인 관리 (manage_data_pipeline)

**언제 참조:** ETL/ELT 워크플로우를 설계하거나 파이프라인 장애를 해결할 때

**협업 대상:**
- product-analytics: 분석 요구사항 수신 — 어떤 데이터를 어떤 형태로 제공해야 하는지
- platform-devops: 파이프라인 인프라 및 스케줄 관리
- backend-engineering: 소스 데이터 스키마 및 변경 알림 수신

**작업 절차:**
1. 배치 처리와 스트리밍 처리를 목적에 맞게 선택한다
2. 각 파이프라인 단계에 데이터 품질 검증 로직을 삽입한다
3. 파이프라인 실패 시 재처리 전략(Checkpoint, 멱등성)을 포함한다
4. 처리 지연과 누락을 실시간으로 감지하는 알림 체계를 구성한다
5. 처리 완료 후 데이터 품질 리포트를 product-analytics에 전달한다

**산출물:** 파이프라인 코드, 데이터 품질 검증 로직, 모니터링 설정, 재처리 전략

**주의사항:**
- 멱등성: 동일 데이터를 여러 번 처리해도 결과가 같아야 한다 — 중복 처리를 방지하는 로직을 반드시 포함한다
- 파이프라인 장애 시 알림이 늦으면 데이터 손실이 발생한다 — 알림 임계값을 엄격하게 설정한다

### 데이터 스키마 관리 (manage_data_schema)

**언제 참조:** 데이터 모델 설계 또는 스키마 변경 계획 수립 시

**협업 대상:**
- backend-engineering: 스키마 변경이 API에 미치는 영향 분석
- product-analytics: 새 스키마가 분석 쿼리에 미치는 영향 확인

**작업 절차:**
1. Schema Evolution 전략(Backward/Forward/Full Compatibility)을 결정한다
2. 하위 호환성을 유지하는 변경만 허용한다 (신규 필드 추가는 가능, 기존 필드 타입 변경/삭제는 버전 업)
3. 신규 필드 추가, 타입 변경, 컬럼 삭제 시 영향 범위를 사전에 분석한다
4. 다운타임 없는 롤링 마이그레이션을 설계한다 (Expand-then-Contract 패턴)
5. 스키마 변경 이력을 버전 관리에 기록한다

**산출물:** 스키마 정의 파일, 마이그레이션 스크립트, 영향 범위 분석 문서

**주의사항:**
- 기존 필드 삭제는 소비자(consumer)가 모두 마이그레이션된 것을 확인한 후에만 실행한다
- 스키마 변경 시 product-analytics와 사전 합의하지 않으면 기존 분석 쿼리가 깨진다
