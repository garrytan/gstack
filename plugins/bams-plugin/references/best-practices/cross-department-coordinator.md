# Cross-Department Coordinator Best Practices

## Responsibility: 부서 간 의존성 관리, 갈등 중재, Phase 전환 핸드오프 조율

---

### 의존성 매핑 (map_dependencies)

**언제 참조:** 복수 부서가 참여하는 Phase 시작 전, 또는 부서 간 블로커 발생 시

**협업 대상:**
- pipeline-orchestrator: 의존성 분석 결과를 병렬화 전략에 반영
- project-governance: 크리티컬 패스 영향 분석 공유

**작업 절차:**
1. 참여 부서 목록과 각 부서의 입력/출력 산출물을 수집한다
2. 산출물 간 의존 관계를 방향 그래프로 표현한다
3. 크리티컬 패스를 식별하고 해당 경로의 부서장에게 우선 자원을 할당하도록 권고한다
4. 순환 의존성이 있으면 즉시 pipeline-orchestrator에게 에스컬레이션한다

**산출물:** 의존성 맵 (입력-출력 관계 그래프, 크리티컬 패스 표시)

**주의사항:**
- 의존성 맵은 팀 모든 에이전트가 이해할 수 있는 수준으로 작성한다
- 암묵적 의존성(API 계약, 데이터 스키마)을 반드시 명시적으로 표현한다

---

### 갈등 중재 (mediate_conflict)

**언제 참조:** 부서 간 우선순위 충돌, 리소스 경합, 해석 차이가 발생했을 때

**협업 대상:**
- pipeline-orchestrator: 중재 결과를 파이프라인 계획에 반영
- product-strategy: 전략적 우선순위 기준 확인

**작업 절차:**
1. 충돌 내용과 각 부서의 입장을 중립적으로 수집한다
2. 사용자 가치와 데이터를 기준으로 판단 근거를 구성한다
3. 충돌 해결 옵션(최소 2개)과 각 옵션의 트레이드오프를 제시한다
4. 합의된 결정을 Decision Log에 기록하고 양 부서에 통보한다
5. 자체 해결 불가 시 pipeline-orchestrator에게 에스컬레이션한다

**산출물:** 갈등 중재 보고서 (충돌 내용, 판단 근거, 결정사항, Decision Log)

**주의사항:**
- 개인 취향이나 권위가 아닌 데이터와 사용자 가치 기준으로 판단한다
- 중재 결과는 당사자 모두가 납득할 수 있는 근거와 함께 제시한다

---

### 핸드오프 관리 (manage_handoff)

**언제 참조:** Phase 전환 시 이전 Phase 부서장의 산출물을 다음 Phase 부서장에게 전달할 때

**협업 대상:**
- 이전 Phase 부서장: 산출물 완결성 및 포맷 정합성 확인
- 다음 Phase 부서장: 전달받은 산출물 수령 확인 및 컨텍스트 이해
- pipeline-orchestrator: 핸드오프 완료 보고

**작업 절차:**
1. 이전 Phase의 필수 산출물 목록을 확인하고 모두 생성되었는지 검증한다
2. 산출물의 포맷 정합성(API 계약, 데이터 스키마, 문서 형식)을 점검한다
3. 다음 Phase 부서장에게 필요한 컨텍스트(배경, 결정 사항, 미해결 이슈)를 정리한다
4. 핸드오프 패키지(산출물 경로 목록 + 컨텍스트 요약)를 구성하여 다음 Phase에 전달한다

**산출물:** 핸드오프 패키지 (산출물 경로 목록, 컨텍스트 요약, 미해결 이슈 목록)

**주의사항:**
- 산출물 누락이 있으면 핸드오프를 중단하고 pipeline-orchestrator에게 NO-GO를 요청한다
- 인터페이스 불일치(API 계약 불일치, 스키마 불일치)는 즉시 해당 부서장에게 수정을 요청한다

---

### 블로커 해소 (resolve_blocker)

**언제 참조:** 한 에이전트의 지연이 다른 에이전트를 블로킹할 때

**협업 대상:**
- pipeline-orchestrator: 블로커 현황 보고 및 우회 경로 제안
- project-governance: 일정 영향도 분석

**작업 절차:**
1. 블로커의 원인(기술적 장애, 의존성 미충족, 의사결정 지연)을 분류한다
2. 블로커가 해소될 때까지 대기 vs. 우회 경로 실행 중 최적 대안을 제안한다
3. 우회 경로가 가능하면 임시 Mock/Stub를 사용하여 블로킹된 에이전트가 병렬로 진행할 수 있도록 조율한다
4. 블로커 해소 후 Mock/Stub를 실제 구현으로 교체하는 통합 절차를 계획한다

**산출물:** 블로커 해소 계획 (원인 분석, 우회 경로, 통합 절차)

**주의사항:**
- 우회 경로 사용 시 Mock/Stub의 존재를 명확히 기록하여 이후 통합 작업이 누락되지 않도록 한다
- 블로커가 3시간 이상 지속되면 pipeline-orchestrator에게 에스컬레이션한다
