# Product Strategy Best Practices

## Responsibility: 제품 비전 정의, 로드맵 우선순위 결정, 이해관계자 정렬, 기획 Phase 부서장

---

### 제품 비전 정의 (define_product_vision)

**언제 참조:** 새 피처 기획 시작 시, 또는 제품 방향성 재정립이 필요할 때

**협업 대상:**
- business-analysis: 기능 명세 도출 위임 — PRD 초안을 입력으로 제공
- ux-research: 사용자 여정 맵핑 위임 — PRD 초안을 입력으로 제공
- project-governance: 일정/리스크 분석 위임 — PRD 초안을 입력으로 제공
- product-analytics: 제품 성과 데이터 기반 전략 검증
- design-director: 디자인 방향 전달, PRD → 디자인 브리프 핸드오프

**작업 절차:**
1. 기존 코드베이스를 Glob, Read로 확인하여 현재 시스템 경계와 기술적 제약을 파악한다
2. 핵심 문제, 타깃 사용자(페르소나), 제공 가치(Value Proposition), 성공 지표(North Star Metric)를 정의한다
3. PRD 초안을 `.crew/artifacts/prd/{slug}.md`에 작성한다
4. business-analysis, ux-research, project-governance에게 동시 위임하여 병렬로 심화 분석을 진행한다
5. 3개 에이전트의 보고를 종합하여 PRD를 확정한다

**산출물:** 확정된 PRD (`/crew/artifacts/prd/{slug}.md`), 기능 명세서, 사용자 여정 맵, 일정/리스크 분석

**주의사항:**
- 비전 문장은 측정 가능한 성공 지표와 반드시 연결되어야 한다
- "모든 사용자"는 타깃이 아니다 — 페르소나 수준으로 구체화한다
- 기술적 실현 가능성을 확인하지 않은 비전은 수립하지 않는다

### 로드맵 우선순위 결정 (prioritize_roadmap)

**언제 참조:** 여러 기능 후보 중 구현 순서를 결정해야 할 때

**협업 대상:**
- business-analysis: 기능별 구현 난이도 및 범위 확인
- project-governance: 일정 제약과 리스크 고려
- business-kpi: 각 기능의 사업 임팩트 수치 확인

**작업 절차:**
1. 기능 후보 목록을 수집한다
2. 각 기능에 대해 RICE(Reach × Impact × Confidence / Effort) 또는 ICE 점수를 계산한다
3. 전략 가치, 구현 난이도, 품질 리스크 세 축으로 평가 매트릭스를 작성한다
4. 의존성 관계를 명시하고 순서가 뒤바뀌면 안 되는 항목을 식별한다
5. "지금 안 하면 안 되는 것"과 "나중에 해도 되는 것"을 명확히 분리한다

**산출물:** 로드맵 우선순위 문서 (기능 목록, 점수 매트릭스, 의존성 맵, 결정 근거)

**주의사항:**
- 감에 의한 판단을 배제하고 정량적 근거를 반드시 첨부한다
- 우선순위 결정 후 이해관계자(product-analytics, business-kpi, design-director)와 정렬한다

### 이해관계자 정렬 (align_stakeholders)

**언제 참조:** 부서 간 해석 차이나 의사결정 충돌이 감지될 때, 또는 Phase 시작 전 전략 정렬 시

**협업 대상:**
- cross-department-coordinator: 부서 간 충돌 중재 협업
- pipeline-orchestrator: 정렬 결과를 파이프라인 계획에 반영

**작업 절차:**
1. 각 부서의 관점과 제약 조건을 먼저 청취하고 요약한다
2. 충돌이 있으면 데이터와 사용자 가치 기준으로 중재한다
3. 합의된 결정을 Decision Log에 이유와 함께 기록한다
4. 미결 항목은 담당자와 기한을 지정하여 방치되지 않도록 관리한다

**산출물:** 이해관계자 정렬 결과 (각 부서 관점 요약, 충돌 지점 및 중재 결과, Decision Log, 미결 항목)

**주의사항:**
- 미결 항목을 "나중에 결정"으로 방치하면 구현 중 충돌이 발생한다 — 반드시 담당자와 기한을 지정한다
- Decision Log는 향후 동일한 논의가 반복되는 것을 방지하는 핵심 산출물이다

### 전략적 트레이드오프 판단 (judge_strategic_tradeoff)

**언제 참조:** 단기 수익 vs. 장기 성장, 기술 부채 수용 vs. 속도 등 전략적 판단이 필요할 때

**협업 대상:**
- business-kpi: 각 옵션의 재무 임팩트 분석
- project-governance: 각 옵션의 일정/리스크 영향 분석
- pipeline-orchestrator: 판단 결과를 파이프라인 방향 조정에 반영

**작업 절차:**
1. 트레이드오프의 축(시간, 비용, 품질, 기술 부채 등)을 명확히 정의한다
2. 각 옵션의 단기/장기 영향을 정량화한다
3. 핵심 가정을 식별하고 가정이 틀릴 때의 리스크를 평가한다
4. 최소 2개 옵션을 제시하고 추천안과 근거를 명시한다

**산출물:** 트레이드오프 판단 문서 (옵션 비교, 핵심 가정, 추천안 및 근거)

**주의사항:**
- "추천안 없음"은 허용되지 않는다 — 데이터가 불충분해도 현재 최선의 추천을 제시해야 한다
- 추천안에 대한 반대 의견과 그에 대한 반론을 함께 기록한다
