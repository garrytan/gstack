# Platform DevOps Best Practices

## Responsibility: 인프라 코드 관리, CI/CD 오케스트레이션, 관측성 및 장애 관리

---

### 인프라 코드 관리 (manage_infrastructure_as_code)

**언제 참조:** 인프라 변경, 새 환경 구성, 또는 인프라 이력 추적이 필요할 때

**협업 대상:**
- backend-engineering: 애플리케이션 요구사항(메모리, CPU, 네트워크 정책) 확인
- release-quality-gate: 인프라 변경이 릴리즈 게이트에 미치는 영향 확인
- data-integration: 데이터 파이프라인 인프라 요구사항 확인

**작업 절차:**
1. 기존 인프라 코드 구조를 Read로 파악하여 컨벤션을 따른다
2. 리소스를 선언적 코드(Terraform, CloudFormation, Pulumi)로 정의한다
3. 환경 간 차이를 변수화하여 dev/staging/prod 설정을 코드로 관리한다
4. 변경 전 `plan/preview` 명령으로 예상 변경 사항을 반드시 확인한다
5. 변경 이력을 버전 관리에 커밋한다

**산출물:** 인프라 코드 파일, 환경별 설정, 변경 계획(plan) 출력

**주의사항:**
- `apply` 전 반드시 `plan`으로 예상 변경 사항을 확인한다 — 리소스 삭제가 포함된 경우 에스컬레이션한다
- 프로덕션 인프라 변경은 maintenance window 내에 진행한다
- 시크릿(비밀번호, API 키)을 코드에 직접 포함하지 않는다 — Secrets Manager를 사용한다

### CI/CD 오케스트레이션 (orchestrate_cicd)

**언제 참조:** 새 파이프라인 구성, 빌드/배포 실패 진단, 또는 배포 전략 변경 시

**협업 대상:**
- automation-qa: 테스트 스위트를 CI 파이프라인에 연동
- release-quality-gate: 배포 게이트 조건(테스트 통과율, 보안 스캔 결과) 합의
- backend-engineering / frontend-engineering: 빌드 요구사항 및 환경 변수 확인

**작업 절차:**
1. 코드 커밋부터 프로덕션 배포까지의 파이프라인을 단계별로 설계한다
2. 각 단계의 게이트 조건을 정의한다 (빌드 성공, 테스트 통과, 보안 스캔 통과)
3. 롤백 전략을 포함한다 (블루-그린, 카나리, 롤링 중 상황에 맞게 선택)
4. 병렬 실행 가능한 단계(테스트 병렬화)를 식별하여 파이프라인 속도를 최적화한다
5. 배포 완료 후 헬스체크를 포함한다

**산출물:** CI/CD 파이프라인 설정 파일, 게이트 조건 정의, 롤백 절차

**주의사항:**
- 카나리 배포 시 트래픽 비율을 단계적으로 증가시킨다 (5% → 25% → 100%)
- 롤백 절차는 사전에 테스트하여 실제 장애 시 즉시 실행 가능하도록 준비한다
- CI 파이프라인 실패 시 원인을 분류한다: flaky test vs. 실제 regression

### 관측성 및 장애 관리 (manage_observability_incidents)

**언제 참조:** 장애 발생 시, 또는 관측성 시스템 구축/개선 시

**협업 대상:**
- backend-engineering: 애플리케이션 로그 포맷 및 메트릭 정의 협의
- data-integration: 이벤트 로그 수집 파이프라인 연동
- release-quality-gate: 배포 후 모니터링 기준치 합의

**작업 절차:**
1. 메트릭(Prometheus), 로그(ELK/Loki), 트레이스(Jaeger/Zipkin) 세 축의 관측성을 구축한다
2. 이상 징후 알림 규칙을 설정한다 (에러율 임계값, 응답 시간 SLA, 리소스 사용률)
3. 장애 발생 시 런북(Runbook)을 실행하여 복구 시간을 최소화한다
4. 장애 대응 후 포스트모텀(Post-Mortem)을 작성하여 재발 방지책을 수립한다

**산출물:** 관측성 설정 코드, 알림 규칙, 런북, 장애 포스트모텀

**주의사항:**
- 알림(alert)이 너무 많으면 alert fatigue가 발생한다 — 액션 가능한 알림만 설정한다
- 런북은 실제 장애 상황에서 5분 내 실행 가능한 수준으로 작성한다
- 포스트모텀은 비난 없는(blameless) 문화로 작성한다 — 사람이 아닌 시스템을 개선한다

### Preflight 권한 체크 (check_preflight_permissions)

**언제 참조:** 모든 위임 작업 수행 전 — 생략 불가

**협업 대상:**
- pipeline-orchestrator: 권한 문제 발생 시 에스컬레이션

**작업 절차:**
1. `disallowedTools` 목록을 확인하여 대상 파일에 Write/Edit 권한이 있는지 검증한다
2. 대상 파일이 `.crew/` 외부인지 확인한다
3. Bash 실행 권한이 필요한지 분석한다
4. 권한 미충족 시 즉시 pipeline-orchestrator에게 에스컬레이션한다 (재시도 0회)

**산출물:** Preflight 체크 결과 (통과/실패, 실패 시 에스컬레이션 메시지)

**주의사항:**
- 이 체크를 생략하면 권한 에러로 재위임이 발생하여 파이프라인이 10분 이상 지연된다
- 권한 에러는 작업을 시도한 후 발견하는 것이 아니라 사전에 예방해야 한다
