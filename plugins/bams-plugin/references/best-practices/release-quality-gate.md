# Release Quality Gate Best Practices

## Responsibility: 출시 준비 상태 검토, 롤백 준비 상태 확인, 출시 후 품질 모니터링, 출시 기준 관리

---

### 출시 준비 상태 검토 (review_release_readiness)

**언제 참조:** 배포 전 GO/NO-GO 판단이 필요할 때

**협업 대상:**
- qa-strategy: 테스트 완료 상태 및 미해결 결함 현황 수신
- platform-devops: 배포 승인 전달 및 롤백 절차 확인
- performance-evaluation: 성능 벤치마크 달성 여부 확인
- product-analytics: 릴리즈 후 모니터링 기준치 확인

**작업 절차:**
1. 다음 항목을 모두 확인한 후에만 GO 판단을 내린다:
   - 모든 P0/P1 기능이 완료되고 검증됨
   - Critical/Major 미해결 결함이 0건 (또는 인수된 예외 사유가 존재)
   - 회귀 테스트 스위트 전체 통과
   - 성능 벤치마크 달성
   - 보안 스캔 통과
2. 각 항목에 대한 근거 데이터를 첨부한다
3. GO 판단 시 platform-devops에게 배포 승인을 전달한다
4. NO-GO 판단 시 구체적인 차단 사유와 해결 기준을 명시한다

**산출물:** 출시 준비 체크리스트 완료 보고서 (항목별 상태, GO/NO-GO 판단, 근거 데이터)

**주의사항:**
- "아마도 괜찮을 것"이라는 판단으로 GO를 내리지 않는다 — 모든 항목이 확인된 후에만 GO이다
- 예외 승인(Critical 결함이 있지만 GO)을 내릴 때는 인수 사유와 후속 조치 계획을 반드시 기록한다

---

### 롤백 준비 상태 확인 (check_rollback_readiness)

**언제 참조:** 배포 전 롤백 절차가 준비되었는지 확인할 때

**협업 대상:**
- platform-devops: 롤백 스크립트 및 이전 버전 아티팩트 보존 확인
- backend-engineering: DB 마이그레이션 역방향 호환성 확인
- data-integration: 피처 플래그 비활성화 가능 여부 확인

**작업 절차:**
1. 롤백 스크립트가 존재하고 최근 테스트되었는지 확인한다
2. DB 마이그레이션이 있는 경우 역방향(Rollback) 마이그레이션이 준비되었는지 확인한다
3. 피처 플래그로 기능을 즉시 비활성화할 수 있는지 확인한다
4. 이전 버전 배포 아티팩트가 보존되어 있는지 확인한다
5. 롤백 실행 시 예상 소요 시간을 확인한다 (목표: 15분 이내)

**산출물:** 롤백 준비 체크리스트 완료 보고서

**주의사항:**
- 롤백 준비가 되지 않은 상태에서는 배포를 진행하지 않는다
- DB 마이그레이션 롤백은 데이터 손실이 발생할 수 있다 — 반드시 백엔드 엔지니어링과 검토한다

---

### 출시 후 품질 모니터링 (monitor_post_release_quality)

**언제 참조:** 배포 완료 후 초기 안정성을 확인할 때 (배포 후 최소 1시간 ~ 24시간)

**협업 대상:**
- platform-devops: 에러율, 응답 시간 모니터링 데이터 수신
- product-analytics: 사용자 행동 이상 탐지 협업
- business-kpi: 핵심 비즈니스 지표 변동 확인

**작업 절차:**
1. 에러율 기준치를 사전에 정의하고 임계값 초과 시 즉시 알림을 설정한다
2. 배포 후 5분, 15분, 1시간, 4시간, 24시간 시점에 주요 지표를 확인한다
3. 에러율이 임계값을 초과하면 즉시 pipeline-orchestrator에게 보고하고 롤백을 검토한다
4. 24시간 모니터링이 안정적으로 완료되면 최종 출시 완료를 선언한다

**산출물:** 배포 후 모니터링 보고서 (시간대별 지표 현황, 이슈 발생 여부, 최종 상태)

**주의사항:**
- 배포 후 자리를 비우지 않는다 — 최소 1시간은 직접 모니터링한다
- 에러율이 급증하면 망설이지 말고 롤백한다 — "조금 더 지켜보자"는 상황을 악화시킨다

### 품질 회고 (conduct_quality_retrospective)

**언제 참조:** 출시 완료 후 게이트 기준의 지속적 개선을 위해

**협업 대상:**
- qa-strategy: 게이트 기준 개선 사항 공유
- pipeline-orchestrator: 회고 데이터 제공

**작업 절차:**
1. 출시 후 발견된 이슈를 분석한다 (릴리즈 후 첫 주 내 발생한 버그)
2. 게이트에서 놓친 이슈의 원인을 파악한다 (체크리스트 미비, 기준치 부적절 등)
3. 게이트 기준을 개선할 항목을 식별하고 qa-strategy와 합의한다
4. 개선 사항을 다음 릴리즈의 게이트 기준에 반영한다

**산출물:** 품질 회고 보고서 (출시 후 발견 이슈, 게이트 미비 원인, 기준 개선 사항)

**주의사항:**
- 게이트 기준 강화는 개발 속도에 영향을 미친다 — 효과 대비 부담을 고려하여 우선순위를 정한다
- "이런 이슈가 다시 발생하지 않도록" 규칙을 추가할 때마다 체크리스트가 길어진다 — 주기적으로 불필요한 항목을 정리한다
