# QA Strategy Best Practices

## Responsibility: 테스트 전략 정의, 테스트 케이스 설계, 품질 리스크 평가, QA Phase 부서장

---

### 테스트 전략 정의 (define_test_strategy)

**언제 참조:** pipeline-orchestrator로부터 검증 Phase 실행 위임을 받았을 때

**협업 대상:**
- automation-qa: 테스트 전략에 따른 자동화 테스트 실행 위임
- product-analytics: 품질 지표 설계 협업
- release-quality-gate: 품질 기준 합의 — 릴리즈 차단 조건 설정
- design-director: 디자인 검증 기준 합의 (비주얼 회귀 테스트)

**작업 절차:**
1. 프로젝트 특성(기술 스택, 릴리스 주기, 변경 규모)을 분석한다
2. 테스트 레벨(단위/통합/E2E)과 유형(기능/성능/보안/비주얼)을 조합한다
3. 위험 기반 우선순위를 적용한다 — 변경 빈도, 복잡도, 외부 의존성이 높은 구간을 집중 테스트한다
4. automation-qa에게 테스트 실행을 위임하고 defect-triage에게 결함 분류를 위임한다
5. QA 리포트를 집계하여 pipeline-orchestrator에게 보고한다

**산출물:** 테스트 전략 문서 (테스트 레벨/유형 선택, 우선순위, 위임 계획), QA 리포트

**주의사항:**
- "모든 것을 테스트한다"는 전략이 아니다 — 위험 기반으로 우선순위를 정한다
- QA 부서장으로서 직접 테스트를 실행하기보다 automation-qa, defect-triage에게 위임하고 결과를 취합한다

### 테스트 케이스 설계 (design_test_cases)

**언제 참조:** business-analysis의 기능 명세를 바탕으로 테스트 케이스를 도출할 때

**협업 대상:**
- business-analysis: 인수 기준 및 업무 규칙 수신
- automation-qa: 자동화 가능한 테스트 케이스 식별 및 전달

**작업 절차:**
1. business-analysis의 기능 명세에서 인수 기준을 추출한다
2. 경계값 분석(BVA), 동치 분할(EP), 상태 전이 등 기법으로 테스트 케이스를 체계적으로 도출한다
3. 요구사항 추적 매트릭스를 작성하여 모든 요구사항이 최소 1개의 테스트 케이스로 커버되는지 확인한다
4. 자동화 가능 여부를 분류하고 automation-qa에게 전달할 케이스를 선별한다

**산출물:** 테스트 케이스 목록 (ID, 전제 조건, 단계, 기대 결과, 자동화 여부), 요구사항 추적 매트릭스

**주의사항:**
- Happy Path만 테스트하면 엣지 케이스에서 버그가 발생한다 — 경계값과 예외 흐름을 반드시 포함한다
- 요구사항 추적 매트릭스 없이 "충분히 테스트했다"고 판단하지 않는다

### 품질 리스크 평가 (assess_quality_risks)

**언제 참조:** 테스트 전략 수립 전, 또는 릴리즈 전 품질 게이트 판단 시

**협업 대상:**
- business-analysis: 기능별 복잡도 및 영향 범위 확인
- defect-triage: 과거 결함 이력 데이터 수신
- release-quality-gate: 고위험 영역 정보를 릴리즈 게이트 기준에 반영

**작업 절차:**
1. 변경 빈도, 복잡도, 외부 의존성, 과거 결함 이력을 기준으로 위험도를 평가한다
2. 위험도가 높은 구간에 더 많은 테스트 자원을 집중 배분한다
3. 위험 매트릭스(발생 가능성 × 영향도)로 우선순위를 정렬한다
4. 고위험 항목은 release-quality-gate에 통보한다

**산출물:** 품질 리스크 평가서 (위험 매트릭스, 집중 테스트 영역, 고위험 항목 목록)

**주의사항:**
- 과거 결함 이력은 미래 위험 지표 중 가장 신뢰할 수 있다 — defect-triage 데이터를 반드시 참조한다
- 위험도가 "낮다"고 판단된 구간도 최소 1개의 스모크 테스트를 포함한다

### 디자인 검증 전략 (verify_design_implementation)

**언제 참조:** Figma 디자인 명세 대비 실제 구현의 시각적 충실도를 검증할 때

**협업 대상:**
- design-director, ui-designer: 디자인 명세 및 검증 기준 합의
- automation-qa: 비주얼 회귀 테스트 자동화 위임

**작업 절차:**
1. design-director와 ui-designer와 함께 검증 기준을 합의한다 (허용 픽셀 오차, 컬러 허용 오차)
2. 검증 대상을 선정한다: 핵심 컴포넌트 상태, 반응형 브레이크포인트, 다크/라이트 모드
3. automation-qa에게 비주얼 회귀 테스트 자동화를 위임한다
4. 시각적 차이 보고서를 검토하고 수정이 필요한 항목을 frontend-engineering에 전달한다

**산출물:** 디자인 검증 계획서 (검증 기준, 대상 컴포넌트 목록), 시각적 차이 보고서

**주의사항:**
- 허용 오차 기준 없이 비주얼 회귀 테스트를 시작하면 오탐(false positive)이 많이 발생한다
- 비주얼 회귀 테스트 기준 스크린샷은 design-director 승인 후 업데이트한다
