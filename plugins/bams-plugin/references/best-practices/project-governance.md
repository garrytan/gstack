# Project Governance Best Practices

## Responsibility: 딜리버리 계획, 리스크 관리, 의존성 관리, 진행 상태 보고

---

### 딜리버리 계획 수립 (plan_delivery)

**언제 참조:** product-strategy로부터 PRD를 받아 구현 일정을 산정할 때

**협업 대상:**
- product-strategy: 로드맵 우선순위와 전략적 판단 확인
- business-analysis: 작업 분해 시 요구사항 범위 확인
- pipeline-orchestrator: 파이프라인 일정 연동
- cross-department-coordinator: 부서 간 의존성 및 블로커 해소

**작업 절차:**
1. 기존 코드베이스를 Glob, Read로 확인하여 기술적 복잡도를 일정 산정에 반영한다
2. 작업 분해 구조(WBS)를 먼저 확정한 후 일정을 산정한다
3. 각 작업의 예상 소요 시간에 불확실성 버퍼(10~30%)를 포함한다
4. 크리티컬 패스를 식별하고 해당 경로에 우선 자원을 배분한다
5. 마일스톤은 검증 가능한 산출물 기준으로 정의한다 ("~완료" 아닌 "~동작 확인")

**산출물:** 딜리버리 계획 (마일스톤 목록, WBS, 크리티컬 패스, 자원 배분, 전제 조건)

**주의사항:**
- 버퍼 없는 일정은 계획이 아닌 희망이다 — 반드시 불확실성 버퍼를 포함한다
- 마일스톤 "~완료"는 허용되지 않는다 — 검증 가능한 완료 기준을 정의한다

### 리스크/의존성 관리 (manage_risk_dependency)

**언제 참조:** 파이프라인 시작 시, 또는 새 리스크가 식별될 때

**협업 대상:**
- cross-department-coordinator: 부서 간 의존성 블로커 해소 협업
- executive-reporter: 리스크 현황을 정기 보고에 포함
- pipeline-orchestrator: 고위험 리스크 발생 시 에스컬레이션

**작업 절차:**
1. 리스크 등록부(Risk Register)를 작성하고 주기적으로 갱신한다
2. 각 리스크에 발생 가능성(H/M/L)과 영향도(H/M/L)를 부여하여 우선순위를 결정한다
3. 리스크마다 트리거 조건과 대응 계획(회피/완화/전가/수용)을 수립한다
4. 의존성은 방향(제공/수신)과 상태(확정/미확정)를 명시한다
5. 외부 의존성(API, 서드파티)은 별도 관리하고 대안 계획을 마련한다

**산출물:** 리스크 등록부 (리스크 목록, 발생 가능성/영향도, 트리거, 대응 전략), 의존성 맵

**주의사항:**
- 리스크가 현실화되면 즉시 pipeline-orchestrator에게 에스컬레이션한다
- 외부 의존성(외부 API, 서드파티 서비스)은 항상 대안 계획을 포함해야 한다

### 진행 상태 보고 (report_status)

**언제 참조:** 정기 보고 주기, 또는 블로커/편차 발생 시

**협업 대상:**
- executive-reporter: 상태 보고 데이터를 대시보드에 반영
- pipeline-orchestrator: 일정 지연 시 에스컬레이션

**작업 절차:**
1. 주관적 표현 대신 수치 기반 지표를 사용한다 (완료율, 잔여 일수 등)
2. 계획 대비 실적 편차가 있으면 원인과 보정 계획을 함께 보고한다
3. 신호등 체계(Green/Yellow/Red)로 전체 상태를 한눈에 파악 가능하도록 요약한다
4. 미결 의사결정 사항과 블로커를 별도 섹션으로 분리한다
5. 다음 보고 시점까지의 예상 진행 사항을 명시한다

**산출물:** 상태 보고서 (신호등 상태, 마일스톤 진행 현황, 블로커, 리스크 변동, 다음 기간 전망)

**주의사항:**
- 일정 지연이 마일스톤에 영향을 미치면 즉시 pipeline-orchestrator에게 에스컬레이션한다
- 보고서에 "잘 진행 중"이라는 표현은 사용하지 않는다 — 수치로 대체한다

### 에스컬레이션 판단 (judge_escalation)

**언제 참조:** 자체 해결 가능한 이슈와 상위 의사결정이 필요한 이슈를 구분할 때

**협업 대상:**
- pipeline-orchestrator: 에스컬레이션 수신 및 대응
- cross-department-coordinator: 부서 간 에스컬레이션 조율

**작업 절차:**
1. 이슈를 자체 해결 가능 vs. 상위 의사결정 필요로 분류한다
2. 에스컬레이션 기준을 적용한다: 일정 영향 > 1일, 리스크 등급 High, 예산 영향 등
3. 에스컬레이션 시 `issue`, `attempted`, `impact`, `options`, `recommendation`을 포함한 메시지를 작성한다
4. 에스컬레이션 결과를 이력으로 기록한다

**산출물:** 에스컬레이션 메시지 (issue, attempted, impact, options 최소 2개, recommendation)

**주의사항:**
- 에스컬레이션 없이 자체 판단으로 해결하다가 실패하면 전체 파이프라인에 영향을 미친다
- "나중에 에스컬레이션하겠다"는 허용되지 않는다 — 기준 초과 즉시 에스컬레이션한다
