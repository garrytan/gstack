# Defect Triage Best Practices

## Responsibility: 결함 분류, 재현 보증, 근본 원인 추적, 우선순위 결정, 결함 라우팅

---

### 결함 분류 (classify_defects)

**언제 참조:** 새 버그가 보고될 때, 또는 QA 테스트에서 발견된 결함을 처리할 때

**협업 대상:**
- qa-strategy: 분류 결과를 QA 리포트에 통합
- backend-engineering / frontend-engineering: 결함 라우팅 후 수정 협업

**작업 절차:**
1. 모든 결함에 동일한 분류 기준을 적용한다 — 보고자의 주관을 보정한다
2. 심각도를 부여한다: Critical(서비스 불능), Major(핵심 기능 장애), Minor(우회 가능), Trivial(외관상 이슈)
3. 결함 유형을 분류한다: 기능, UI, 성능, 보안, 접근성
4. 영향 범위를 정의한다: 전체 사용자 영향 vs. 특정 조건에서만 발생
5. 중복 결함을 식별하여 기존 이슈와 병합하거나 연결한다

**산출물:** 분류된 결함 목록 (심각도, 유형, 영향 범위, 중복 여부)

**주의사항:**
- 분류 결과에 근거를 명시한다 — "왜 Critical인가"를 설명할 수 없으면 분류가 잘못된 것이다
- 보고자가 "Critical"이라고 했어도 기준에 맞지 않으면 재분류한다 — 분류의 일관성이 중요하다

### 재현 보증 (assure_reproduction)

**언제 참조:** 결함을 담당자에게 라우팅하기 전 재현 가능성을 확보할 때

**협업 대상:**
- 결함 보고자: 재현에 필요한 추가 정보 수집
- automation-qa: 재현 가능한 결함을 자동화 회귀 테스트에 추가

**작업 절차:**
1. 재현 단계를 단계별로 명시한다 (전제 조건, 수행 단계, 실제 결과, 기대 결과)
2. 재현 환경 조건을 기록한다 (OS, 브라우저 버전, 화면 크기, 로그인 상태, 테스트 데이터)
3. 재현율을 측정한다 (10회 중 몇 회 발생하는가)
4. 재현율이 낮은 결함은 가능한 원인 조건을 좁혀가며 추가 조사한다

**산출물:** 재현 보고서 (재현 단계, 환경 조건, 재현율, 스크린샷/로그 첨부)

**주의사항:**
- 재현 단계가 불명확한 결함은 라우팅하지 않는다 — 개발자가 독립적으로 재현 가능해야 한다
- "100% 재현됨"과 "가끔 발생"을 명확히 구분한다 — 재현율에 따라 디버깅 접근이 다르다

### 근본 원인 추적 (track_root_cause)

**언제 참조:** 동일한 유형의 결함이 반복 발생할 때, 또는 릴리즈 후 결함 패턴을 분석할 때

**협업 대상:**
- qa-strategy: 근본 원인 분석 결과를 테스트 전략 개선에 반영
- backend-engineering / frontend-engineering: 구조적 원인 제거 협업

**작업 절차:**
1. 최근 N개 결함에서 패턴을 식별한다 (동일 모듈, 동일 개발자 코드, 동일 기능 유형)
2. "왜?"를 5번 반복하여(5 Whys) 표면적 원인이 아닌 구조적 원인을 파악한다
3. 단일 버그 수정이 아닌 구조적 원인 제거 방안을 제안한다
4. 재발 방지를 위한 회귀 테스트 추가를 automation-qa에게 요청한다

**산출물:** 근본 원인 분석 보고서 (패턴 식별, 5 Whys 분석, 구조적 원인, 재발 방지 방안)

**주의사항:**
- "개발자 실수"를 근본 원인으로 결론내리지 않는다 — 실수를 유발한 시스템/프로세스 문제를 찾는다
- 근본 원인 분석 결과는 pipeline-orchestrator 회고에 필수 입력 자료로 제공한다

### 결함 라우팅 (route_defects)

**언제 참조:** 분류된 결함을 수정 담당자에게 배분할 때

**협업 대상:**
- frontend-engineering: UI, 클라이언트 로직 결함 라우팅
- backend-engineering: API, 서버 로직, 데이터 결함 라우팅
- platform-devops: 인프라, 배포, 환경 관련 결함 라우팅
- data-integration: 이벤트 수집, 데이터 파이프라인 결함 라우팅

**작업 절차:**
1. 결함의 발생 위치를 분석한다 (프론트엔드, 백엔드, 인프라, 데이터)
2. 결함 특성에 따라 적합한 에이전트에 라우팅한다
3. 라우팅 메시지에 재현 단계, 환경 조건, 기대 결과, 우선순위를 포함한다
4. Critical 결함은 즉시 라우팅하고 담당자 확인을 요청한다

**산출물:** 결함 라우팅 완료 목록 (결함 ID, 라우팅 대상, 우선순위, 라우팅 시각)

**주의사항:**
- "어디 문제인지 모르겠다"며 여러 에이전트에게 동시 라우팅하지 않는다 — 분석 후 단일 담당자에게 라우팅한다
- Critical/Major 결함 라우팅 후 24시간 내 수정 착수 여부를 확인한다
