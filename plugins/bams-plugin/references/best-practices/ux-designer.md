# UX Designer Best Practices

## Responsibility: 와이어프레임 설계, 프로토타이핑, 접근성 검증, A/B 테스트 설계, 사용자 플로우 정의

---

### 와이어프레임 설계 (wireframing)

**언제 참조:** ux-research의 페르소나/여정 맵을 받아 화면 구조를 설계할 때

**협업 대상:**
- ux-research: 페르소나와 사용자 여정 맵 수신 — 반드시 먼저 확인
- ui-designer: 완성된 와이어프레임을 고충실도 UI 설계의 입력으로 전달
- frontend-engineering: 인터랙션 플로우의 기술적 구현 가능성 사전 확인

**작업 절차:**
1. ux-research의 페르소나와 사용자 여정 맵을 반드시 먼저 확인한다
2. Lo-fi 와이어프레임으로 정보 구조와 인터랙션 흐름을 시각화한다
3. 각 화면에 화면 ID, 진입 조건, 나가는 경로를 주석으로 표시한다
4. 빈 상태(데이터 없음), 로딩 상태, 에러 상태를 모든 화면에 반드시 설계한다
5. design-director 피드백 후 Hi-fi 프로토타입으로 발전시킨다

**산출물:** Lo-fi 와이어프레임 (모든 화면, 상태, 분기 포함), Hi-fi 프로토타입

**주의사항:**
- 페르소나와 여정 맵 없이 와이어프레임을 시작하지 않는다 — 사용자 목표 기반 설계가 우선이다
- 엣지 케이스(빈 상태, 에러, 권한 없음)를 처음부터 포함한다 — 나중에 추가하면 레이아웃이 무너진다

---

### 사용자 플로우 정의 (define_user_flow)

**언제 참조:** 새 피처의 사용자 경로를 설계하거나 기존 플로우를 개선할 때

**협업 대상:**
- ux-research: 여정 맵에서 주요 경로와 이탈 지점 수신
- business-analysis: 업무 규칙과 상태 전이 확인 (플로우 분기 조건)
- frontend-engineering: 플로우의 기술적 구현 가능성 확인

**작업 절차:**
1. 모든 엔트리 포인트를 식별한다 (직접 접근, 이메일 링크, 검색 등)
2. 결정 분기(조건에 따른 경로 분리)를 플로우 다이어그램으로 명시한다
3. 성공 상태와 실패 상태를 모두 정의한다
4. 엣지 케이스(권한 없음, 세션 만료, 네트워크 오류)를 포함한다
5. 각 단계에서 사용자에게 명확한 피드백이 있는지 확인한다

**산출물:** 사용자 플로우 다이어그램 (모든 경로, 분기, 엣지 케이스 포함)

**주의사항:**
- 플로우에서 "뒤로 가기" 경로를 빠뜨리는 경우가 많다 — 모든 단계에서 취소/이전 경로를 명시한다
- 플로우 다이어그램은 frontend-engineering의 클라이언트 플로우 구현의 직접 입력으로 사용된다

### 접근성 검증 (accessibility_audit)

**언제 참조:** 설계된 와이어프레임 또는 프로토타입의 접근성 문제를 검토할 때

**협업 대상:**
- ui-designer: 접근성 문제(명도 대비, 터치 타겟 크기)를 고충실도 UI에 반영
- frontend-engineering: 스크린 리더 호환성, 키보드 네비게이션 구현 가이드 제공
- qa-strategy: 접근성 테스트 케이스 협의

**작업 절차:**
1. WCAG 2.2 AA 기준으로 명도 대비를 검증한다 (일반 텍스트 4.5:1, 대형 텍스트 3:1)
2. 포커스 관리를 검토한다 (포커스 순서, 포커스 표시자 가시성)
3. 스크린 리더 호환성을 확인한다 (ALT 텍스트, ARIA 레이블, 의미론적 마크업)
4. 키보드 네비게이션으로 모든 인터랙션이 가능한지 확인한다
5. 각 문제에 심각도를 부여하고 구체적인 수정 방안을 제시한다

**산출물:** 접근성 검증 보고서 (문제 목록, 심각도, WCAG 기준, 수정 방안)

**주의사항:**
- 접근성은 릴리즈 후 추가 작업이 아니라 설계 단계에서 해결해야 한다
- 심각도 "치명적" 항목은 release-quality-gate의 배포 차단 조건으로 통보한다

### A/B 테스트 설계 (design_ab_test)

**언제 참조:** UX 가설을 실험으로 검증하거나 전환율 개선 실험을 계획할 때

**협업 대상:**
- product-analytics: 성공 지표 및 측정 방식 합의
- experimentation: 실험 설계(표본 크기, 통계적 검정력) 협업
- frontend-engineering / backend-engineering: 실험 코드(피처 플래그) 구현 협업

**작업 절차:**
1. UX 가설을 명확히 정의한다 ("CTA 버튼 색상을 파란색에서 주황색으로 바꾸면 클릭률이 높아질 것이다")
2. 성공 지표(주 지표 1개)와 가드레일 지표를 정의한다
3. A(대조군)와 B(실험군)의 차이를 최소화한다 — 변수를 1개로 제한한다
4. experimentation에게 표본 크기와 실험 기간 산정을 요청한다
5. 실험 코드(피처 플래그) 구현 스펙을 frontend-engineering에게 전달한다

**산출물:** A/B 테스트 설계서 (가설, 성공 지표, 실험군/대조군 정의, 표본 크기 요청)

**주의사항:**
- 주 지표는 반드시 1개로 한정한다 — 여러 지표를 동시에 최적화하면 해석이 불가능해진다
- 실험 종료 조건을 사전에 정의한다 — 중간에 임의로 종료하면 통계적 유의성이 무효가 된다
