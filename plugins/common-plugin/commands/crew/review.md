# Crew Review

Crew 오케스트레이터로서 5개 전문 병렬 에이전트를 활용한 다관점 코드 리뷰를 실행합니다.

리뷰 대상: $ARGUMENTS

## 코드 최신화

Bash로 `git rev-parse --is-inside-work-tree 2>/dev/null`를 실행하여 git 저장소인지 확인합니다.

**git 저장소인 경우**: Bash로 `git branch --show-current`를 실행하여 현재 브랜치를 확인한 뒤, `git pull origin {현재 브랜치}`를 실행하여 원격 저장소의 최신 코드를 가져옵니다. 충돌이 발생하면 사용자에게 알리고 중단합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵합니다.

## 리뷰 범위 결정

**$ARGUMENTS가 비어있거나 제공되지 않은 경우:**
1. Bash로 `git diff` 실행하여 미스테이지 변경사항 확인
2. 미스테이지 변경이 없으면, `git diff --cached`로 스테이지된 변경사항 확인
3. 변경사항이 전혀 없으면, 사용자에게 무엇을 리뷰할지 물어보고 중단

**$ARGUMENTS가 파일이나 디렉토리를 지정한 경우:**
- Glob으로 존재 여부 검증
- 해당 파일/디렉토리를 리뷰

**$ARGUMENTS가 "pr" 또는 PR 번호인 경우:**
- Bash로 `git diff main...HEAD` (또는 적절한 베이스 브랜치)를 실행하여 PR diff 획득
- `git diff --name-only main...HEAD`로 변경된 파일 목록도 확보

## 사전 조건

`.crew/config.md`가 있으면 읽습니다 (프로젝트 컨벤션 참조). `CLAUDE.md`가 있으면 읽습니다.

## Phase 1: 파일 수집

1. 리뷰 범위의 모든 파일 식별
2. 각 파일 읽기 (15개 초과 시 우선순위: 변경된 파일 → 핵심 로직 → 생성/벤더 파일 스킵)
3. git 변경사항 리뷰 시, diff 출력을 컨텍스트로 캡처

## Phase 2: 5개 리뷰 에이전트 병렬 실행

**Task tool**을 사용하여 5개 에이전트를 동시에 실행 (subagent_type: **"Explore"**, model: **"sonnet"** — 읽기 전용 분석, 버그 감지 품질 중요):

**리뷰할 파일**: [파일 경로 목록]
[파일 내용 또는 diff 포함]
**프로젝트 가이드라인**: [CLAUDE.md가 있으면 참조]

분석 항목:
1. 로직 오류, off-by-one 에러, null/undefined 처리
2. 누락되거나 잘못된 에러 핸들링과 에러 전파

## Phase 3: 리뷰 종합

5개 에이전트가 모두 반환한 후:

1. 모든 에이전트의 발견 사항 수집
2. **중복 제거**: 여러 에이전트가 같은 이슈를 발견한 경우 (같은 파일, 같은 라인, 같은 개념) 병합하고 가장 높은 신뢰도 유지
3. **충돌 감지**: 이슈 간 trade-off 또는 race condition 관계를 식별 (아래 상세)
4. **정렬**: Critical → Major → Minor 순. 같은 심각도 내에서는 신뢰도 높은 순
5. 심각도별 총 이슈 수 **집계**

### 충돌 감지

중복 제거 후, 다음 기준으로 이슈 간 충돌을 탐지합니다:

**Trade-off 감지 기준:**
- 같은 파일/영역에 대해 서로 반대 방향의 수정을 제안하는 이슈 쌍
  - 예: Performance "캐싱 추가" vs Security "민감 데이터 캐시 금지"
  - 예: Performance "인라인 처리" vs Quality "함수 분리"
- 서로 다른 카테고리(Security, Performance, Quality 등)에서 같은 코드를 다루는 이슈

**Race condition 감지 기준:**
- 두 이슈의 Fix(수정안)가 같은 파일의 같은 라인 범위(±5줄)를 수정하는 경우
- 한 이슈의 Fix를 적용하면 다른 이슈의 Fix가 무효화되는 경우

**충돌이 감지되면:**

Architect 중재 에이전트를 실행합니다 (Task tool, subagent_type: **"Explore"**, model: **"sonnet"**):

> **시니어 소프트웨어 아키텍트**로서 코드 리뷰에서 발견된 충돌 이슈들을 중재합니다.
>
> **충돌 이슈 쌍**: [충돌하는 이슈 쌍 전문 삽입 — 각 이슈의 카테고리, 파일, 설명, 수정안 포함]
> **해당 파일의 현재 코드**: [Read로 해당 파일의 관련 코드 영역 읽기]
> **프로젝트 컨텍스트**: [CLAUDE.md와 .crew/config.md가 있으면 참조]
>
> 각 충돌 쌍에 대해:
> 1. 충돌의 본질을 분석 (trade-off인지 race condition인지)
> 2. 양쪽 관점의 타당성을 평가
> 3. 두 관점을 모두 만족시키는 **통합 해결안**을 제시
> 4. 통합이 불가능하면, 각 선택지의 장단점을 명시하고 권장안을 제시
>
> 반환 형식:
> ```
> ## 충돌 [N]: [이슈A] ↔ [이슈B]
>
> **충돌 유형**: Trade-off | Race Condition
> **분석**: [왜 충돌하는지 설명]
>
> **통합 해결안**: [두 관점을 모두 충족하는 방안]
> **통합 수정 코드**: [구체적 코드 제안]
>
> 또는 (통합 불가 시):
> **선택지 A**: [이슈A의 수정안 적용]
>   - 장점: [...]
>   - 단점: [...]
> **선택지 B**: [이슈B의 수정안 적용]
>   - 장점: [...]
>   - 단점: [...]
> **권장**: [A 또는 B] — [이유]
> ```

Architect 에이전트 반환 후, 충돌 이슈 쌍을 **충돌 그룹**으로 묶어 관리합니다:
- 통합 해결안이 있으면: 원래 두 이슈를 하나의 통합 이슈로 대체
- 통합 불가면: 두 이슈를 충돌 그룹으로 태깅하고, 선택지와 권장안을 첨부

## Phase 4: 리포트 저장

타임스탬프 slug 생성 (예: `2026-02-13-143052`).

리뷰 리포트를 `.crew/artifacts/review/review-[timestamp].md`에 저장합니다. 다음 섹션을 포함:
- **요약**: 심각도별 이슈 건수 테이블, 리뷰 파일 수
- **Critical/Major/Minor 이슈**: 각 이슈 상세 (카테고리-N, file:line, 설명, 수정안)
- **충돌 이슈**: 각 충돌별 유형, 관련 이슈 쌍, Architect 판정, 해결안
- **긍정적 관찰**: 잘된 점 기록

## Phase 5: 결과 제시

사용자에게 결과를 표시합니다: 리뷰 범위, 리포트 경로, 심각도별 이슈 건수, 충돌 건수.

그 다음 모든 Critical 및 Major 이슈를 상세와 함께 나열합니다.

충돌 이슈가 있으면, 일반 이슈 뒤에 별도로 표시합니다 (충돌 유형, 관련 이슈, Architect 제안).

## Phase 6: 이슈 수정 제안

Critical 또는 Major 이슈가 없으면 이 Phase를 스킵합니다.

Critical 또는 Major 이슈가 있으면, **AskUserQuestion** 도구로 수정할 이슈를 선택받습니다.

**multiSelect: true** 질문을 구성합니다:

Question: "수정할 이슈를 선택하세요."
Header: "Fix Issues"
Options (multiSelect: true) — 발견된 Critical/Major 이슈에서 동적으로 구성:

각 이슈를 옵션으로 변환합니다:
- **label**: `[CATEGORY-N] file:line` (예: `[SECURITY-1] src/auth.ts:42`)
- **description**: 이슈 설명 요약 + 심각도 (예: `"Critical: SQL 인젝션 취약점 — 사용자 입력 미검증"`)

옵션 수가 4개를 초과하면, 여러 번의 AskUserQuestion으로 나누어 물어봅니다:
- 첫 질문: 이슈 1-4
- 두 번째 질문: 이슈 5-8
- 계속...
각 질문의 Header에 `"Fix (1/N)"` 형식으로 현재 페이지를 표시합니다.

사용자가 "Other"를 선택한 경우, 입력된 텍스트에서 이슈 번호를 파싱합니다.

사용자가 아무것도 선택하지 않으면 (모든 질문에서 선택 없음), 수정을 스킵합니다.

### 충돌 이슈 처리

충돌 그룹이 있으면, 일반 이슈 선택 후 충돌 이슈를 별도로 처리합니다.

각 충돌 그룹에 대해 **AskUserQuestion** (multiSelect: false):

**통합 해결안이 있는 경우:**

Question: "충돌 [N]: [이슈A] ↔ [이슈B] — 어떻게 처리할까요?"
Header: "Conflict"
Options:
- **통합안 적용** - "[Architect의 통합 해결안 요약]"
- **이슈 A만** - "[이슈A 수정안 요약]"
- **이슈 B만** - "[이슈B 수정안 요약]"
- **스킵** - "이 충돌은 수동으로 처리"

**통합 불가인 경우:**

Question: "충돌 [N]: [이슈A] ↔ [이슈B] — 어느 쪽을 우선할까요?"
Header: "Conflict"
Options:
- **선택지 A (권장)** - "[이슈A 수정안] — 장점: [...], 단점: [...]"
- **선택지 B** - "[이슈B 수정안] — 장점: [...], 단점: [...]"
- **스킵** - "이 충돌은 수동으로 처리"

### 선택된 이슈 수정

사용자가 선택한 이슈들에 대해:

1. 각 이슈의 제안된 수정사항(Fix)을 Edit 도구로 직접 적용
2. 수정 적용 시 기존 코드를 깨뜨리지 않도록 주의
3. 수정된 각 이슈를 리뷰 리포트(`.crew/artifacts/review/review-[timestamp].md`)에서 `[수정됨]` 태그 추가

수정 완료 후 요약을 표시합니다: 수정된 이슈 목록 (✓), 충돌 해결 목록, 미수정(스킵) 이슈 목록.

### 수정 변경사항 확인

git 저장소인 경우 (Bash로 `git rev-parse --is-inside-work-tree` 확인), 수정 적용 후 변경사항을 사용자에게 보여주고 확인받습니다.

1. Bash로 `git diff --stat`을 실행하여 변경 요약 표시
2. Bash로 `git diff`를 실행하여 상세 diff 표시

사용자에게 표시:
```
리뷰 수정 완료 — 변경사항 확인
──────────────────────────────────────
[git diff --stat 출력]
```

**AskUserQuestion**으로 확인:

Question: "수정 결과를 적용할까요?"
Header: "Confirm"
Options:
- **적용** - "변경사항을 유지"
- **되돌리기** - "모든 수정을 되돌리기"
- **부분 되돌리기** - "특정 파일만 되돌리기"

**적용** 선택 시: 진행.

**되돌리기** 선택 시:
- `git checkout -- [수정된 파일들]` 실행
- 리뷰 리포트에서 `[수정됨]` 태그를 제거
- "수정이 되돌려졌습니다." 메시지 출력

**부분 되돌리기** 선택 시:
- 수정된 파일 목록을 AskUserQuestion (multiSelect)으로 선택
- 선택된 파일만 `git checkout -- [file]` 실행

수정된 파일이 남아있으면 제안: "수정된 파일에 대해 `/crew:review`를 재실행하여 검증할 수 있습니다."
