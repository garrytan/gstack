# Crew Q

Crew 오케스트레이터로서 코드베이스에 대한 질문에 답변합니다. 질문에서 범위를 자동 감지하여 관련 코드를 수집한 뒤 프로젝트 컨텍스트에 기반한 답변을 제공합니다.

입력: $ARGUMENTS

$ARGUMENTS가 비어있으면, 사용자에게 무엇이 궁금한지 물어보고 중단합니다.

## Phase 1: 사전 조건

Glob으로 `.crew/config.md`가 존재하는지 확인합니다. 없으면:
- 출력: "Crew가 초기화되지 않았습니다. `/crew:init`을 실행하여 설정하세요."
- 여기서 중단.

`.crew/config.md`를 읽어 프로젝트 컨텍스트를 로드합니다.

## Phase 2: 범위 자동 감지

$ARGUMENTS를 분석하여 질문의 범위를 자동으로 판별합니다. 다음 3가지 카테고리 중 하나로 라우팅합니다:

### 판별 규칙

**카테고리 A — 파일 기반**: $ARGUMENTS에 파일 경로 패턴이 포함된 경우
- 슬래시(`/`)가 포함된 경로 (예: `src/auth/middleware.ts`)
- 확장자가 포함된 파일명 (예: `handler.go`, `utils.py`)
- 판별되면 → Phase 3A로 진행

**카테고리 B — 함수/메서드 기반**: $ARGUMENTS에 함수나 메서드 식별자가 포함된 경우
- camelCase, snake_case, PascalCase 패턴의 식별자 (예: `handleAuth`, `parse_token`, `UserService`)
- 괄호가 붙은 호출 형태 (예: `processPayment()`)
- 클래스/메서드 표기 (예: `AuthService.validate`, `auth_module::verify`)
- 판별되면 → Phase 3B로 진행

**카테고리 C — 기능 설명 기반**: 위 패턴에 해당하지 않는 자연어 설명
- "결제 흐름이 어떻게 되어있어?", "인증 로직 설명해줘" 등
- 판별되면 → Phase 3C로 진행

**복합 케이스**: 여러 카테고리에 걸치는 경우 (예: "src/auth/middleware.ts의 handleAuth 함수가 왜 이렇게 구현되어있어?")
- 가장 구체적인 카테고리를 우선합니다: A > B > C
- 파일 경로가 있으면 A, 파일 경로 없이 함수명이 있으면 B

**판별 불가**: 질문이 너무 짧거나 모호하면 **AskUserQuestion**으로 확인:

Question: "질문의 범위를 좁히면 더 정확한 답변이 가능합니다. 어떻게 찾을까요?"
Header: "Scope"
Options:
- **파일 지정** - "특정 파일 경로를 알려주세요"
- **함수/클래스 지정** - "함수명이나 클래스명을 알려주세요"
- **자동 탐색** - "질문 내용을 바탕으로 관련 코드를 자동으로 찾기"

## Phase 3A: 파일 기반 코드 수집

$ARGUMENTS에서 파일 경로를 추출합니다.

1. Glob으로 해당 파일이 존재하는지 확인합니다.
   - **존재하면**: Read로 파일 내용을 읽습니다.
   - **존재하지 않으면**: Glob으로 유사한 경로를 검색하고, 결과가 있으면 사용자에게 "이 파일을 의미하셨나요?" 형태로 확인합니다. 결과가 없으면 사용자에게 알리고 중단합니다.

2. 파일이 너무 큰 경우 (500줄 초과): $ARGUMENTS의 질문 키워드와 관련된 섹션만 추출합니다.

→ Phase 4로 진행 (수집된 코드 + 파일 경로를 컨텍스트로 전달)

## Phase 3B: 함수/메서드 기반 코드 수집

$ARGUMENTS에서 함수/메서드/클래스 식별자를 추출합니다.

서브에이전트 실행 (Task tool, subagent_type: **"Explore"**, model: **"haiku"**):

> **코드 검색 전문가**로서 코드베이스에서 특정 함수/메서드/클래스를 찾고 관련 컨텍스트를 수집합니다.
>
> **검색 대상**: [추출된 식별자]
> **프로젝트 컨텍스트**: [.crew/config.md의 아키텍처 요약]
>
> 수행 작업:
> 1. Grep으로 식별자의 **정의(definition)** 위치를 검색 (함수 선언, 클래스 선언, 메서드 선언)
> 2. 정의가 포함된 파일을 Read로 읽어 해당 함수/메서드의 전체 코드 추출
> 3. 같은 파일 내의 관련 타입, 인터페이스, 상수도 포함
> 4. Grep으로 이 함수/메서드의 **호출자(caller)** 를 최대 5개 검색
> 5. 호출자의 관련 코드 스니펫 수집 (호출 라인 ±5줄)
>
> 반환 내용:
> - **정의 위치**: file:line
> - **함수/메서드 전체 코드**: 정의부터 끝까지
> - **관련 타입/인터페이스**: 있으면 포함
> - **주요 호출자**: 최대 5개 (file:line + 스니펫)
> - **못 찾은 경우**: 유사한 이름의 후보 목록

서브에이전트가 식별자를 찾지 못한 경우:
- 유사 후보가 있으면 사용자에게 제시하고 선택 요청
- 후보도 없으면 사용자에게 알리고 중단

→ Phase 4로 진행 (수집된 코드 + 정의 위치 + 호출자를 컨텍스트로 전달)

## Phase 3C: 기능 설명 기반 코드 수집

서브에이전트 실행 (Task tool, subagent_type: **"Explore"**, model: **"sonnet"**):

> **시니어 소프트웨어 아키텍트**로서 기능 설명을 바탕으로 관련 코드를 탐색합니다.
>
> **기능 설명**: [$ARGUMENTS에서 질문 내용]
> **프로젝트 컨텍스트**: [.crew/config.md 전체 내용]
>
> 수행 작업:
> 1. 기능 설명에서 핵심 키워드 추출
> 2. Grep과 Glob으로 관련 파일 탐색 (파일명, 디렉토리명, 코드 내 키워드)
> 3. 발견된 파일 중 가장 관련성 높은 파일 최대 5개를 Read로 읽기
> 4. 파일 간의 관계 (import/호출 체인) 추적
> 5. 해당 기능의 데이터 흐름과 핵심 로직 파악
>
> 반환 내용:
> - **관련 파일**: 관련성 순으로 정렬된 파일 목록 (file:line — 역할 설명)
> - **핵심 코드**: 기능의 핵심 로직이 담긴 코드 스니펫 (각 파일에서)
> - **데이터 흐름**: 이 기능이 시스템에서 어떻게 동작하는지 요약
> - **관련 설정/의존성**: 환경 변수, 설정 파일, 외부 서비스 연동 등

→ Phase 4로 진행 (수집된 코드 + 관련 파일 + 데이터 흐름을 컨텍스트로 전달)

## Phase 4: 답변

수집된 코드와 프로젝트 컨텍스트를 기반으로 질문에 답변합니다.

서브에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"sonnet"**):

> **시니어 소프트웨어 엔지니어**로서 코드베이스에 대한 질문에 정확하게 답변합니다.
>
> **질문**: [$ARGUMENTS 전체]
> **프로젝트 컨텍스트**: [.crew/config.md — 프로젝트 개요, 아키텍처, 컨벤션]
> **수집된 코드**: [Phase 3에서 수집된 모든 코드와 메타데이터]
>
> 답변 방침:
> 1. **코드 근거 기반**: 추측하지 않고 수집된 코드를 근거로 답변. 코드 참조 시 file:line 형태로 출처를 명시.
> 2. **구조화된 답변**: 질문 유형에 따라 적절한 구조 사용:
>    - "왜 이렇게 되어있어?" → 설계 의도, 히스토리, 트레이드오프 설명
>    - "어떻게 동작해?" → 실행 흐름, 데이터 흐름 단계별 설명
>    - "이 파일/함수 뭐야?" → 역할, 책임, 주요 인터페이스 설명
>    - "어디서 ~하고 있어?" → 관련 코드 위치와 스니펫 제시
> 3. **프로젝트 맥락 반영**: 프로젝트의 아키텍처와 컨벤션을 고려한 답변
> 4. **간결성**: 핵심을 먼저 말하고 상세는 뒤에. 불필요한 서론 없이 바로 답변.
> 5. **불확실성 표시**: 코드만으로 판단이 어려운 부분은 명시적으로 "추정" 또는 "확인 필요"로 표시
>
> 반환 형식:
> ```
> ## 답변
> [핵심 답변 — 1-3문장]
>
> ### 상세
> [코드 참조를 포함한 상세 설명]
>
> ### 관련 코드
> [file:line 형태의 참조 목록]
> ```

서브에이전트 반환 후, 답변을 사용자에게 표시합니다.

## Phase 5: Q&A 히스토리 저장

`.crew/artifacts/qa/` 디렉토리가 없으면 Bash `mkdir -p`로 생성합니다.

질문에서 slug를 생성합니다 (핵심 키워드 2-3개, 소문자, 하이픈 연결).

`.crew/artifacts/qa/[slug]-qa.md`에 저장합니다:

```markdown
# Q&A: [질문 요약 한 줄]

> 날짜: [현재 ISO timestamp]
> 범위: [카테고리 A/B/C — 파일/함수/기능]
> 대상: [파일 경로 또는 식별자 또는 기능 키워드]

## 질문

[$ARGUMENTS 전체]

## 답변

[Phase 4의 답변 전체]

## 참조 코드

[관련 파일:라인 목록]
```

**같은 slug의 파일이 이미 존재하면**: 기존 파일에 `---` 구분선 후 새 Q&A를 **추가(append)** 합니다. 같은 주제에 대한 질문이 누적됩니다.

저장 후 사용자에게 표시:
```
Q&A 저장됨: .crew/artifacts/qa/[slug]-qa.md
```
