# Crew Dev

Crew 오케스트레이터로서 완전한 개발 파이프라인을 실행합니다: 기획 → 구현 → 리뷰 → 테스트.

입력: $ARGUMENTS

$ARGUMENTS가 비어있으면, 사용자에게 무엇을 개발할지 물어보고 중단합니다.

## 코드 최신화

Bash로 `git rev-parse --is-inside-work-tree 2>/dev/null`를 실행하여 git 저장소인지 확인합니다.

**git 저장소인 경우**: Bash로 `git branch --show-current`를 실행하여 현재 브랜치를 확인한 뒤, `git pull origin {현재 브랜치}`를 실행하여 원격 저장소의 최신 코드를 가져옵니다. 충돌이 발생하면 사용자에게 알리고 중단합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵합니다.

## 사전 조건

Glob으로 `.crew/config.md`가 존재하는지 확인합니다. 없으면:
- 출력: "Crew가 초기화되지 않았습니다. `/crew:init`을 실행하여 설정하세요."
- 여기서 중단.

`.crew/config.md`와 `.crew/board.md`를 읽습니다.

## 작업 범위 결정

**$ARGUMENTS가 태스크 ID인 경우 (예: "TASK-001"):**
- `.crew/board.md`에서 해당 태스크 찾기
- 찾지 못하면 사용자에게 알리고 중단
- 의존성이 모두 `## Done` 섹션에 있는지 확인
- 의존성이 충족되지 않으면 블로킹 태스크 목록을 표시하고 중단
- 이 단일 태스크로 Phase 2 (구현)로 진행

**$ARGUMENTS가 board.md에 태스크가 있는 피처 slug인 경우:**
- 해당 피처의 모든 태스크 수집 (`**Feature**: [slug]` 매칭)
- 해당 태스크들로 Phase 2 (구현)로 진행

**$ARGUMENTS가 새로운 피처 설명인 경우:**
- $ARGUMENTS에서 slug를 생성합니다 (소문자, 공백→하이픈, 특수문자 제거, 영문 변환)
- Glob으로 `.crew/artifacts/prd/*-prd.md`와 `.crew/artifacts/design/*-design.md`를 검색합니다
- **slug가 일치하는 PRD와 설계 문서가 모두 존재하면** (이미 `/crew:plan`으로 기획됨):
  - 두 아티팩트를 읽어서 컨텍스트로 사용
  - board.md에서 해당 피처의 태스크를 수집 (`**Feature**: [slug]` 매칭)
  - 사용자에게 알림: "기존 계획 발견: [slug] (PRD + 설계 + 태스크 [N]건). 기존 계획을 사용합니다."
  - Phase 2 (구현)로 진행
- **아티팩트가 없으면** → Phase 1 (기획)으로 진행

## Phase 1: 기획 (기존 계획이 없는 경우만)

전체 기획 워크플로를 인라인으로 실행합니다:

### 1a. PM 에이전트

서브에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"opus"**):

> **시니어 프로덕트 매니저**로서 이 피처 요청을 분석하고 간결한 PRD를 작성합니다.
>
> **피처**: [$ARGUMENTS 삽입]
> **프로젝트 컨텍스트**: [.crew/config.md 내용 삽입]
>
> 작성 내용: 문제 정의, 기능 요구사항 (번호 매김, MUST/SHOULD/MAY 사용), 사용자 스토리 (3-5개), 인수 기준, 범위 외, 미결 질문.
>
> 먼저 Glob과 Read를 사용하여 코드베이스를 파악합니다.

**미결 질문이 있으면** 사용자에게 제시하고 답변을 기다립니다.

### 1b. Architect 에이전트 (2개 병렬)

2개 서브에이전트를 동시에 실행 (Task tool, subagent_type: "general-purpose", model: **"sonnet"**):

**에이전트 A - 코드베이스 분석:**
> **시니어 아키텍트**로서 다음 구현을 위한 코드베이스를 분석합니다: [피처].
> 요구사항: [1a의 PRD]
> 관련 파일 찾기, 아키텍처 매핑, 재사용 가능한 코드와 테스트 패턴 식별.
> 반환: 아키텍처 요약, 핵심 파일 경로, 재사용 가능 코드, 테스트 패턴.

**에이전트 B - 설계 제안:**
> **시니어 아키텍트**로서 다음의 구현을 설계합니다: [피처].
> 요구사항: [1a의 PRD], 프로젝트 컨텍스트: [.crew/config.md]
> 반환: 아키텍처 개요, 파일 계획 (테이블), 인터페이스 정의, 데이터 흐름, 의존성, 위험요소.
> 최소한으로 유지 — 과도한 엔지니어링 없이.

### 1c. 태스크 분해

두 Architect 결과를 종합합니다. 다음을 포함한 태스크 생성:
- 명확한 범위 (구체적 파일)
- 역할 할당 (Developer/Reviewer/QA)
- 우선순위 (P0/P1/P2)
- 의존성
- 인수 기준

### 1d. 아티팩트 저장

1. PRD를 `.crew/artifacts/prd/[slug]-prd.md`에 저장
2. 설계를 `.crew/artifacts/design/[slug]-design.md`에 저장
3. 태스크를 `.crew/board.md`의 `## Backlog`에 추가 (CLAUDE.md 컨벤션에 따른 형식)
4. `.crew/config.md`의 `last_task_id` 업데이트

사용자에게 계획 요약을 제시합니다. 질문: "구현을 진행할까요?"
사용자가 아니라고 하면, 여기서 중단합니다. 태스크는 보드에 저장되어 나중에 사용할 수 있습니다.

## Phase 1.5: Git 체크포인트

구현을 시작하기 전에 현재 코드 상태를 보존합니다.

먼저 Bash로 `git rev-parse --is-inside-work-tree`를 실행하여 git 저장소인지 확인합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵하고 Phase 2로 진행합니다.

**git 저장소인 경우**: Bash로 `git status --porcelain`을 실행하여 미커밋 변경사항이 있는지 확인합니다.

**AskUserQuestion**으로 체크포인트 방식을 선택받습니다:

Question: "구현 시작 전 코드를 어떻게 보존할까요?"
Header: "Git"
Options:
- **Feature branch** - "새 브랜치를 생성하여 작업 (예: crew/[slug])"
- **Stash** - "현재 변경사항을 stash하고 현재 브랜치에서 작업"
- **스킵** - "체크포인트 없이 바로 진행"

**Feature branch** 선택 시:
1. 미커밋 변경사항이 있으면 먼저 `git stash`
2. `git checkout -b crew/[slug]` 실행 (slug는 피처명에서 생성)
3. stash가 있으면 `git stash pop`
4. 사용자에게 알림: "브랜치 `crew/[slug]` 생성됨. 작업 완료 후 merge하세요."

**Stash** 선택 시:
1. `git stash push -m "crew: checkpoint before [slug]"` 실행
2. 사용자에게 알림: "체크포인트 생성됨. 되돌리려면: `git stash pop`"

선택된 체크포인트 방식을 기억하여 이후 변경 확인 단계에서 참조합니다.

## Phase 2: 구현

board.md에서 태스크 목록을 의존성 순서로 정렬합니다. 배치로 그룹화:
- **배치 1**: 의존성 없는 태스크 (모두 병렬 실행 가능)
- **배치 2**: 의존성이 모두 배치 1에 있는 태스크
- **배치 N**: 모든 태스크가 스케줄될 때까지 계속

각 배치의 모든 태스크를 처리합니다:

### Developer 태스크의 경우:

같은 배치의 여러 Developer 태스크가 파일 겹침이 없으면, Task tool (subagent_type: "general-purpose", model: **"opus"**)을 사용하여 **병렬로** 실행합니다. 그렇지 않으면 순차 실행.

각 Developer 에이전트에게 전달:

> **전문 개발자**로서 이 태스크를 정확하게 구현합니다.
>
> **태스크**: [제목과 설명]
> **생성/수정할 파일**: [태스크에서]
> **기술 설계**: [설계 문서의 관련 섹션]
> **코드베이스 컨벤션**: [CLAUDE.md와 .crew/config.md에서]
> **인수 기준**: [태스크에서]
>
> 코드 작성 방침 (반드시 준수):
> - **단순성 우선**: 요청된 것만 구현. 추상화, 유연성, 불필요한 에러 핸들링 금지. 50줄이면 될 것을 200줄로 만들지 않기.
> - **외과적 변경**: 관련 없는 코드/주석/포맷 개선 금지. 기존 스타일 따르기. 내 변경으로 생긴 미사용 코드만 정리.
> - **목표 기반 실행**: 인수 기준을 체크리스트로 삼아 각 항목을 충족하며 구현.
>
> 지침:
> 1. 먼저 범위 내 기존 파일 읽기
> 2. 기존 코드 패턴을 정확히 따르기
> 3. 완전하게 구현 — TODO나 플레이스홀더 없음
> 4. 지정된 범위 외 파일을 수정하지 않기
> 5. 모든 변경된 줄이 태스크 요구사항에 직접 연결되는지 검증
> 6. 반환: 생성/수정된 파일과 변경 사항 요약

각 Developer 에이전트 반환 후:
- 파일을 읽어 올바르게 생성/수정되었는지 확인
- board.md에서 해당 태스크를 `## Backlog`에서 `## In Review`로 이동
- `> Last updated:` 타임스탬프 업데이트

### 구현 변경사항 확인

git 저장소인 경우, 각 배치의 구현이 완료된 후 변경사항을 사용자에게 보여주고 확인받습니다.

1. Bash로 `git diff --stat`을 실행하여 변경 요약 표시
2. Bash로 `git diff`를 실행하여 상세 diff 표시 (너무 길면 `--stat`만)

사용자에게 표시:
```
구현 완료 — 변경사항 확인
──────────────────────────────────────
[git diff --stat 출력]
```

**AskUserQuestion**으로 확인:

Question: "구현 결과를 적용할까요?"
Header: "Confirm"
Options:
- **적용** - "변경사항을 유지하고 다음 단계로 진행"
- **되돌리기** - "모든 변경사항을 되돌리고 중단"
- **부분 되돌리기** - "특정 파일만 되돌리기"

**적용** 선택 시: 다음 단계로 진행.

**되돌리기** 선택 시:
- `git checkout -- .` 실행하여 모든 변경 되돌리기
- 새로 생성된 파일은 `git clean -fd` 대신 수동으로 삭제 확인
- board.md에서 태스크를 `## Backlog`으로 복원
- 사용자에게 "변경사항이 되돌려졌습니다." 메시지 출력 후 중단

**부분 되돌리기** 선택 시:
- 변경된 파일 목록을 보여주고 되돌릴 파일을 AskUserQuestion (multiSelect)으로 선택받기
- 선택된 파일에 대해 `git checkout -- [file]` 실행
- 다음 단계로 진행

## Phase 2.5: 테스트 코드 생성

현재 배치의 모든 Developer 태스크가 완료된 후, 수정/생성된 파일에 대한 테스트가 존재하는지 확인합니다.

### Step 1: 기존 테스트 확인

Developer 에이전트가 수정/생성한 각 파일에 대해 해당 테스트 파일 검색:
- Glob 사용: `**/*test*`, `**/*spec*`, `**/*_test.*`, `**/*.test.*`
- 수정된 파일을 import하거나 참조하는 테스트 파일이 있는지 확인

### Step 2: 테스트 작성 여부 묻기 (항상)

테스트 커버리지가 없는 수정/생성된 파일이 있으면, **AskUserQuestion** 사용:

Question: "수정된 코드에 대한 테스트 코드를 작성할까요?"
Header: "Tests"
Options:
- **Yes** - "수정된 모든 파일에 대해 테스트 작성"
- **Select** - "테스트할 파일을 선택"
- **Skip** - "이번에는 테스트 스킵"

사용자가 "Skip"을 선택하면 Phase 3으로 진행합니다.

### Step 3: 테스트 디렉토리 결정 (처음에만)

`.crew/config.md` 프론트매터를 읽습니다. `test_dir` 필드를 확인합니다.

**`test_dir`가 null인 경우** (처음):

**AskUserQuestion** 사용:

Question: "테스트 파일을 어디에 작성할까요?"
Header: "Test Dir"
Options:
- **Colocated** - "소스 파일 옆에 작성 (e.g., src/auth/handler_test.go, src/auth/handler.spec.ts)"
- **Separate dir** - "별도 테스트 디렉토리에 작성 (e.g., tests/, __tests__/, test/)"
- **Detect** - "기존 테스트 파일 위치를 따라감"

"Separate dir" 선택 시: 디렉토리 경로를 물어봅니다 (예: `tests/`, `__tests__/`, `test/`).
"Detect" 선택 시: 기존 테스트 파일을 검색하여 패턴을 추론합니다.
"Colocated" 선택 시: 테스트 파일을 소스 파일 옆에 배치합니다.

선택된 패턴을 `.crew/config.md` 프론트매터의 `test_dir`에 저장:
- `test_dir: colocated` — 소스 옆 배치
- `test_dir: tests/` (또는 해당 경로) — 별도 디렉토리
- `test_dir: detect` — 자동 감지

**`test_dir`가 이미 설정된 경우**: 물어보지 않고 저장된 설정을 사용합니다.

### Step 4: 테스트 작성

QA 에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"sonnet"**):

> **시니어 QA 엔지니어**로서 최근 구현된 코드에 대한 테스트를 작성합니다.
>
> **테스트할 파일**: [테스트 커버리지가 없는 파일 목록]
> **테스트 위치**: [test_dir 설정에서 — 테스트 파일을 어디에 생성할지 설명]
> **요구사항**: [PRD에서]
> **테스트 패턴**: [코드베이스 또는 .crew/config.md 컨벤션에서 감지]
> **인수 기준**: [태스크에서]
>
> 지침:
> 1. 이 프로젝트에서 사용하는 테스트 프레임워크 식별
> 2. 각 새로운/수정된 public 함수에 대한 유닛 테스트 작성
> 3. 컴포넌트 간 상호작용에 대한 통합 테스트 작성 (해당되는 경우)
> 4. 엣지 케이스, 에러 시나리오, 경계값 커버
> 5. 기존 테스트 컨벤션을 정확히 따르기
> 6. 테스트 위치 설정에 따라 테스트 파일 배치
> 7. 반환: 생성된 테스트 파일 목록, 커버리지 요약

QA 에이전트 반환 후:
- 테스트 계획을 `.crew/artifacts/test/[slug]-tests.md`에 저장
- 프로젝트에 테스트 러너가 있으면, Bash로 테스트 실행을 시도하고 결과 보고

## Phase 3: 리뷰

모든 구현 태스크 완료 후, **3개 Reviewer 에이전트를 병렬로 실행** (Task tool, subagent_type: **"Explore"**, model: **"sonnet"** — 읽기 전용 분석, 버그 감지 품질 중요):

### Reviewer 1 - 정확성:

> **정확성 중심의 시니어 코드 리뷰어**로서 리뷰합니다.
> **변경된 파일**: [이 피처에서 수정/생성된 모든 파일 목록]
> 위 목록의 각 파일을 읽습니다.
> **요구사항**: [PRD에서]
>
> 점검 항목: 로직 오류, 누락된 엣지 케이스, 잘못된 에러 핸들링, 레이스 컨디션, 미충족 요구사항.
> 각 이슈별: file:line, 심각도 (Critical/Major/Minor), 신뢰도 (0-100), 설명, 수정안.
> 신뢰도 >= 80%인 이슈만 보고.

### Reviewer 2 - 보안 및 성능:

> **보안과 성능 중심의 시니어 코드 리뷰어**로서 리뷰합니다.
> **변경된 파일**: [목록]
> 위 목록의 각 파일을 읽습니다.
>
> 점검 항목: 인젝션 결함, 인증 이슈, 데이터 노출, 하드코딩된 시크릿, N+1 쿼리, 메모리 누수, 블로킹 작업, 비효율적 알고리즘.
> 각 이슈별: file:line, 심각도, 신뢰도 (0-100), 설명, 수정안.
> 신뢰도 >= 80%인 이슈만 보고.

### Reviewer 3 - 코드 품질:

> **코드 품질과 표준 중심의 시니어 코드 리뷰어**로서 리뷰합니다.
> **변경된 파일**: [목록]
> 위 목록의 각 파일을 읽습니다.
> **프로젝트 컨벤션**: [CLAUDE.md와 .crew/config.md에서]
>
> 점검 항목: DRY 위반, 네이밍 이슈, 과도한 복잡성, 스타일 비일관성, 미사용 코드.
> 각 이슈별: file:line, 심각도, 신뢰도 (0-100), 설명, 수정안.
> 신뢰도 >= 80%인 이슈만 보고.

### 리뷰 결과 처리

1. 모든 발견 사항 수집, 중복 제거, 심각도 후 신뢰도 순 정렬
2. 리뷰 리포트를 `.crew/artifacts/review/[slug]-review.md`에 저장

**Critical 이슈 발견 시:**
- 사용자에게 제시
- 수정: Edit 도구로 제안된 수정사항 적용
- 수정된 항목을 리뷰 리포트에서 해결됨으로 이동

**Major 이슈 발견 시:**
- 사용자에게 제시
- 질문: "이 이슈들을 수정할까요? (yes/no)"
- yes이면 수정 적용

수정 후, 리뷰 리포트를 업데이트합니다.

## Phase 3.5: PM Quality Gate (최대 3회 반복)

리뷰 완료 후, PM 에이전트가 최종 품질을 검증합니다. 이 단계는 **최대 3회까지 반복**할 수 있습니다.

현재 반복 횟수를 `iteration = 1`로 초기화합니다.

### Quality Gate 루프 시작

PM Quality Gate 에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"opus"**):

> **시니어 프로덕트 매니저**로서 구현 결과물의 최종 품질을 검증합니다.
>
> **PRD (인수 기준)**: [PRD 전문 삽입]
> **기술 설계**: [설계 문서 요약 삽입]
> **구현된 파일**: [Developer가 생성/수정한 모든 파일 목록]
> **리뷰 리포트**: [리뷰 결과 전문 삽입]
> **테스트 결과**: [테스트 실행 결과가 있으면 삽입]
> **현재 반복**: [iteration]/3
>
> 다음을 검증합니다:
>
> 1. **인수 기준 충족 여부**: PRD의 각 인수 기준을 하나씩 대조하여 구현 상태 확인
>    - 각 기준별로: ✓ 충족 / ✗ 미충족 / △ 부분 충족
> 2. **리뷰 이슈 해결 여부**: Critical/Major 이슈가 모두 적절히 수정되었는지 확인
> 3. **코드 완성도**: TODO, 플레이스홀더, 미구현 부분이 남아있지 않은지 확인
> 4. **설계 준수 여부**: 기술 설계서의 아키텍처와 인터페이스를 따르고 있는지 확인
>
> 구현된 파일들을 직접 Read 도구로 읽어서 검증합니다.
>
> 판정 결과를 다음 형식으로 반환합니다:
>
> ```
> ## Quality Gate 판정
>
> **결과**: PASS | FAIL
>
> ### 인수 기준 검증
> | # | 기준 | 상태 | 비고 |
> |---|------|------|------|
> | 1 | [기준] | ✓/✗/△ | [설명] |
>
> ### 미해결 리뷰 이슈
> [있으면 나열, 없으면 "없음"]
>
> ### 재작업 요청 (FAIL인 경우만)
> **재개발 필요**:
> - [파일: 구체적 수정 사항]
>
> **재리뷰 필요**:
> - [재리뷰가 필요한 영역과 이유]
> ```

### Quality Gate 결과 처리

**PASS인 경우:**
- 사용자에게 "PM Quality Gate 통과 (반복 [iteration]/3)" 메시지 표시
- Quality Gate 판정 결과를 `.crew/artifacts/review/[slug]-review.md`에 추가
- board.md에서 태스크를 `## In Review`에서 `## Done`으로 이동
- Phase 4로 진행

**FAIL인 경우:**
- `iteration`을 1 증가
- 사용자에게 Quality Gate 실패 내용을 표시:
  ```
  PM Quality Gate 실패 (반복 [iteration-1]/3)
  ──────────────────────────────────────
  미충족 인수 기준: [N]건
  미해결 리뷰 이슈: [N]건
  재개발 필요: [파일 목록]
  재리뷰 필요: [영역 목록]
  ```

- **iteration <= 3이면** 재작업 실행:

  **재개발이 필요한 경우** — Developer 에이전트 재실행 (Task tool, subagent_type: "general-purpose", model: **"opus"**):
  > **전문 개발자**로서 PM의 피드백을 반영하여 코드를 수정합니다.
  >
  > **PM 피드백**: [Quality Gate의 재작업 요청 내용 전문]
  > **수정할 파일**: [재개발 필요 파일 목록]
  > **원래 인수 기준**: [PRD에서]
  > **코드베이스 컨벤션**: [CLAUDE.md와 .crew/config.md에서]
  >
  > Phase 2의 Developer 에이전트와 동일한 코드 작성 방침을 준수합니다. 추가 지침:
  > 1. PM이 지적한 미충족 사항만 정확히 해결
  > 2. 기존에 통과한 부분을 깨뜨리지 않도록 주의
  > 3. 반환: 수정된 파일과 변경 사항 요약

  **재리뷰가 필요한 경우** — Reviewer 에이전트 재실행 (Task tool, subagent_type: **"Explore"**, model: **"sonnet"**):
  > PM이 지적한 영역에 대해 집중 리뷰합니다.
  >
  > **재리뷰 영역**: [PM이 지정한 재리뷰 필요 영역]
  > **변경된 파일**: [재개발로 수정된 파일 목록]
  >
  > 해당 영역에 대해서만 정확성, 보안, 성능, 품질을 집중 점검합니다.
  > 신뢰도 >= 80%인 이슈만 보고.

  재작업 완료 후 → **Quality Gate 루프 시작**으로 돌아가서 PM에게 재검증 요청

- **iteration > 3이면** (3회 반복 초과):
  - 사용자에게 경고 표시:
    ```
    PM Quality Gate 3회 반복 후에도 미통과
    ──────────────────────────────────────
    남은 미충족 사항:
      [목록]

    수동 확인이 필요합니다.
    ```
  - 미통과 내역을 `.crew/artifacts/review/[slug]-review.md`에 기록
  - board.md에서 태스크를 `## In Review`에서 `## Done`으로 이동 (미통과 사항 주석 포함)
  - Phase 4로 진행

## Phase 4: 마무리

### README.md 동기화

프로젝트 루트에 `README.md`가 존재하는지 Glob으로 확인합니다.

**`README.md`가 존재하면:**

이번 피처에서 생성/수정된 파일 목록과 PRD를 기반으로, README.md에 영향을 줄 수 있는 변경이 있는지 판단합니다:
- 새로운 API 엔드포인트, CLI 명령, 공개 함수/클래스 추가
- 설정 방법, 환경변수, 의존성 변경
- 사용법, 실행 방법 변경
- 프로젝트 구조 변경 (새 디렉토리, 주요 파일 추가/삭제)
- 기술 스택 변경

**영향이 있으면** `README.md`를 Read로 읽고, 변경사항에 맞춰 해당 섹션을 Edit으로 업데이트합니다. 기존 README.md의 스타일과 구조를 유지합니다.

**`README.md`가 없거나 영향이 없으면:** 이 단계를 스킵합니다.

### 보드 및 히스토리 업데이트

1. 완료된 모든 태스크를 board.md의 `## Done`으로 이동
2. 완료된 태스크를 `.crew/history.md`에 타임스탬프와 함께 추가:
   ```
   ### [ISO date]
   - **TASK-NNN**: [title] (Feature: [slug])
   ```
3. board.md의 `> Last updated:` 업데이트

최종 요약을 제시합니다: 피처명, 생성/수정 파일 목록, 테스트 파일 목록, 리뷰 이슈 요약 (심각도별 건수), QG 결과 (PASS/FAIL, 반복 횟수), 아티팩트 경로, 완료 태스크 수.

git 사용 시 `git add`와 `/crew:sprint status` 안내.
