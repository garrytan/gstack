# Crew Debug

Crew 오케스트레이터로서 버그 조사 → 원인 분석 → 영향 분석 → 수정 → 검증 파이프라인을 실행합니다.

입력: $ARGUMENTS

$ARGUMENTS가 비어있으면, 사용자에게 어떤 버그를 조사할지 물어보고 중단합니다.

## 코드 최신화

Bash로 `git rev-parse --is-inside-work-tree 2>/dev/null`를 실행하여 git 저장소인지 확인합니다.

**git 저장소인 경우**: Bash로 `git branch --show-current`를 실행하여 현재 브랜치를 확인한 뒤, `git pull origin {현재 브랜치}`를 실행하여 원격 저장소의 최신 코드를 가져옵니다. 충돌이 발생하면 사용자에게 알리고 중단합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵합니다.

## 사전 조건

Glob으로 `.crew/config.md`가 존재하는지 확인합니다. 없으면:
- 출력: "Crew가 초기화되지 않았습니다. `/crew:init`을 실행하여 설정하세요."
- 여기서 중단.

`.crew/config.md`와 `.crew/board.md`를 읽습니다.

## Phase 1: 버그 접수 및 재현

### 1a. 버그 정보 파싱

$ARGUMENTS에서 다음을 추출합니다:
- **증상**: 무엇이 잘못되었는가?
- **에러 메시지**: 스택 트레이스, 에러 로그 (있으면)
- **재현 조건**: 어떤 상황에서 발생하는가? (있으면)

정보가 불충분하면 **AskUserQuestion**으로 추가 정보를 요청:

Question: "버그에 대해 추가 정보가 필요합니다."
Header: "Bug Info"
Options:
- **에러 메시지 있음** - "스택 트레이스나 에러 로그를 붙여넣기"
- **재현 방법 있음** - "재현 단계를 설명"
- **증상만 있음** - "현재 정보로 조사 시작"

### 1b. Reproducer 에이전트

서브에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"sonnet"**):

> **시니어 QA 엔지니어**로서 보고된 버그를 분석하고 재현을 시도합니다.
>
> **버그 리포트**: [$ARGUMENTS + 추가 정보 삽입]
> **프로젝트 컨텍스트**: [.crew/config.md 내용 삽입]
>
> 수행 작업:
> 1. 에러 메시지/스택 트레이스가 있으면 파싱하여 관련 파일과 라인 식별
> 2. Glob과 Grep으로 에러 메시지, 함수명, 관련 키워드를 코드베이스에서 검색
> 3. 관련 파일을 읽어 버그 발생 가능 지점 식별
> 4. 테스트 코드가 있으면 읽어서 이 시나리오가 커버되는지 확인
> 5. 재현 가능한 경우: 최소 재현 단계를 정리
> 6. 재현 불가한 경우: 가장 유력한 발생 조건을 추정
>
> 반환 내용:
> - **관련 파일**: 버그와 관련된 파일 목록 (경로:라인)
> - **재현 단계**: 재현 방법 (가능한 경우) 또는 추정 조건
> - **초기 가설**: 버그의 원인에 대한 1-3개 가설
> - **기존 테스트 커버리지**: 이 영역에 대한 테스트 유무

Reproducer 반환 후, 결과를 사용자에게 간략히 표시:
```
버그 조사 시작
──────────────────────────────────────
증상: [요약]
관련 파일: [N]개 식별
초기 가설: [N]개
테스트 커버리지: [있음/없음]
```

## Phase 1.5: Git 체크포인트

수정을 시작하기 전에 현재 코드 상태를 보존합니다.

먼저 Bash로 `git rev-parse --is-inside-work-tree`를 실행하여 git 저장소인지 확인합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵하고 Phase 2로 진행합니다.

**git 저장소인 경우**: Bash로 `git status --porcelain`을 실행하여 미커밋 변경사항이 있는지 확인합니다.

**AskUserQuestion**으로 체크포인트 방식을 선택받습니다:

Question: "수정 시작 전 코드를 어떻게 보존할까요?"
Header: "Git"
Options:
- **Feature branch** - "새 브랜치를 생성하여 작업 (예: crew/fix-[slug])"
- **Stash** - "현재 변경사항을 stash하고 현재 브랜치에서 작업"
- **스킵** - "체크포인트 없이 바로 진행"

**Feature branch** 선택 시:
1. 미커밋 변경사항이 있으면 먼저 `git stash`
2. `git checkout -b crew/fix-[slug]` 실행
3. stash가 있으면 `git stash pop`
4. 사용자에게 알림: "브랜치 `crew/fix-[slug]` 생성됨."

**Stash** 선택 시:
1. `git stash push -m "crew: checkpoint before fix-[slug]"` 실행
2. 사용자에게 알림: "체크포인트 생성됨. 되돌리려면: `git stash pop`"

## Phase 2: 원인 분석 (Root Cause Analysis)

**3개 Investigator 에이전트를 병렬로 실행** (Task tool, subagent_type: **"Explore"**, model: **"sonnet"**):

### Investigator 1 — 코드 흐름 추적:

> **시니어 디버거**로서 버그의 실행 흐름을 추적합니다.
>
> **버그 정보**: [Phase 1 결과: 관련 파일, 재현 단계, 초기 가설]
> **관련 파일**: [Reproducer가 식별한 파일 목록]
>
> 수행 작업:
> 1. 버그 발생 지점부터 역방향으로 코드 흐름 추적 (caller → callee 체인)
> 2. 데이터가 어디서 오염/누락/변형되는지 식별
> 3. 조건 분기에서 예상과 다르게 흐르는 경로 확인
> 4. 관련 함수의 입출력 계약(contract) 위반 여부 확인
>
> 반환 내용:
> - **실행 경로**: 버그 트리거 → 에러 발생까지의 호출 체인
> - **오염 지점**: 데이터가 잘못되기 시작하는 정확한 위치
> - **근본 원인 후보**: 가장 유력한 원인과 근거

### Investigator 2 — 변경 이력 분석:

> **시니어 디버거**로서 최근 변경 이력에서 버그 원인을 찾습니다.
>
> **버그 정보**: [Phase 1 결과]
> **관련 파일**: [목록]
>
> 수행 작업:
> 1. 관련 파일들의 git log 확인 (최근 20개 커밋)
> 2. git blame으로 버그 관련 코드 영역의 최근 변경자/변경일 확인
> 3. 최근 변경이 버그를 도입했을 가능성 평가
> 4. 관련 PR/커밋 메시지에서 의도된 변경인지 확인
>
> 반환 내용:
> - **의심 커밋**: 버그를 도입했을 가능성이 높은 커밋 (SHA, 날짜, 메시지)
> - **변경 전후 비교**: 해당 커밋의 diff에서 버그 관련 부분
> - **회귀 여부**: 이전에 동작하던 것이 깨진 것인지, 원래부터 있던 버그인지

### Investigator 3 — 엣지 케이스 및 환경 분석:

> **시니어 디버거**로서 엣지 케이스와 환경 요인을 조사합니다.
>
> **버그 정보**: [Phase 1 결과]
> **관련 파일**: [목록]
>
> 수행 작업:
> 1. 관련 코드의 경계 조건 분석 (null, 빈 배열, 0, 음수, 최대값)
> 2. 동시성 이슈 확인 (레이스 컨디션, 데드락, 공유 상태)
> 3. 환경 의존성 확인 (설정 파일, 환경 변수, 외부 서비스)
> 4. 타입/인터페이스 불일치 확인
> 5. 에러 핸들링 누락 또는 잘못된 에러 전파 확인
>
> 반환 내용:
> - **엣지 케이스**: 버그를 트리거하는 경계 조건
> - **동시성 이슈**: 발견된 레이스 컨디션 또는 공유 상태 문제
> - **환경 요인**: 설정/환경 의존적 이슈
> - **근본 원인 후보**: 가장 유력한 원인과 근거

### 원인 종합

3개 Investigator가 모두 반환한 후:

1. 각 에이전트의 근본 원인 후보를 수집
2. 일치하는 후보가 있으면 신뢰도 높음으로 태깅
3. **근본 원인을 하나로 확정** (여러 후보가 있으면 증거가 가장 강한 것 선택)
4. 확정이 어려우면 사용자에게 후보를 제시하고 선택 요청

사용자에게 표시:
```
원인 분석 완료
──────────────────────────────────────
근본 원인: [한 줄 요약]
위치: [file:line]
유형: [로직 오류 / 엣지 케이스 / 레이스 컨디션 / 회귀 / 환경 / 타입 불일치]
신뢰도: [높음/중간/낮음]
증거: [일치한 Investigator 수]/3
```

## Phase 3: 영향 분석 (Impact Analysis)

수정 전에 사이드이펙트를 사전 분석합니다.

### 3a. Impact Analyzer 에이전트

서브에이전트 실행 (Task tool, subagent_type: **"Explore"**, model: **"sonnet"**):

> **시니어 소프트웨어 아키텍트**로서 버그 수정의 영향 범위를 분석합니다.
>
> **근본 원인**: [Phase 2에서 확정된 원인]
> **수정 대상 파일**: [file:line]
> **프로젝트 컨텍스트**: [.crew/config.md 내용]
>
> 수행 작업:
> 1. 수정 대상 함수/메서드의 **모든 호출자(caller)** 추적 (Grep으로 함수명 검색, LSP findReferences 사용)
> 2. 수정 대상이 export/public이면 **외부 의존자** 식별
> 3. 수정 대상의 **반환값/사이드이펙트**에 의존하는 코드 식별
> 4. 이 코드의 **현재 동작에 의존하는 다른 코드**가 있는지 확인 ("버그에 의존하는 코드")
> 5. 관련 **테스트 파일** 식별 (수정으로 깨질 수 있는 테스트)
>
> 반환 내용:
> - **직접 영향**: 수정 대상을 직접 호출하는 코드 (파일:라인 목록)
> - **간접 영향**: 호출 체인을 통해 영향받는 코드
> - **버그 의존 코드**: 현재 (잘못된) 동작에 맞춰진 코드 (있으면)
> - **영향받는 테스트**: 수정으로 변경/실패할 수 있는 테스트 파일
> - **위험 등급**: Low / Medium / High — 수정의 폭발 반경 평가
> - **권장 수정 범위**: 수정해야 할 파일 목록 (최소 범위)

### 3b. 범위 잠금 (Scope Lock)

Impact Analyzer 결과를 기반으로 **수정 허용 파일 목록**을 확정합니다:

1. Impact Analyzer가 권장한 수정 범위를 사용자에게 표시:

```
영향 분석 완료
──────────────────────────────────────
위험 등급: [Low/Medium/High]
직접 영향: [N]개 파일
간접 영향: [N]개 파일
버그 의존 코드: [있음 N개 / 없음]

수정 허용 범위:
  ✎ [file1:line] — [수정 이유]
  ✎ [file2:line] — [수정 이유]
  ⚠ [file3:line] — 버그 의존 코드, 함께 수정 필요

관련 테스트: [N]개
```

2. **버그 의존 코드**가 발견되면 경고:
   ```
   ⚠ 주의: 현재 (잘못된) 동작에 의존하는 코드가 [N]개 있습니다.
   이 코드도 함께 수정하지 않으면 버그 수정 후 새로운 문제가 발생합니다.
   ```

3. **위험 등급이 High**이면 사용자에게 확인:

   **AskUserQuestion**:
   Question: "수정 영향 범위가 넓습니다. 계속 진행할까요?"
   Header: "Risk"
   Options:
   - **진행** - "분석된 범위로 수정 진행"
   - **축소** - "핵심 파일만 수정 (버그 의존 코드는 나중에)"
   - **중단** - "수동으로 처리"

이 단계에서 확정된 **수정 허용 파일 목록**을 Phase 4에서 Developer 에이전트에게 전달합니다.

## Phase 4: 수정

### 4a. Developer 에이전트 — 외과적 패치

서브에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"opus"**):

> **전문 개발자**로서 버그를 최소 범위로 수정합니다.
>
> **근본 원인**: [Phase 2 결과]
> **수정 허용 파일**: [Phase 3에서 확정된 목록 — 이 파일만 수정 가능]
> **버그 의존 코드**: [함께 수정해야 할 코드 목록]
> **영향 분석**: [직접/간접 영향 요약]
> **코드베이스 컨벤션**: [CLAUDE.md와 .crew/config.md에서]
>
> 코드 작성 방침 (반드시 준수):
> - **단순성 우선**: 버그 수정에 필요한 최소한의 변경만. 리팩토링 금지. 개선 금지.
> - **외과적 변경**: 수정 허용 파일 목록 외의 파일을 절대 수정하지 않기. 관련 없는 코드 건드리지 않기.
> - **목표 기반 실행**: 근본 원인을 직접 해결하는 변경만. 증상 우회(workaround)가 아닌 원인 제거.
>
> 지침:
> 1. 수정 허용 파일만 읽고 수정 — **목록 외 파일 수정 금지**
> 2. 근본 원인을 직접 해결하는 최소 변경 적용
> 3. 버그 의존 코드가 있으면 함께 수정
> 4. 기존 코드 스타일을 정확히 따르기
> 5. 수정과 무관한 코드 변경 금지 (포맷팅, 네이밍, 주석 개선 등)
> 6. 반환: 수정된 파일, 각 파일의 변경 내용, 수정이 근본 원인을 해결하는 이유

### 4b. 범위 위반 검증

Developer 에이전트 반환 후, 수정된 파일 목록을 Phase 3의 수정 허용 파일 목록과 대조합니다.

**허용 범위를 벗어난 파일이 있으면:**
- 사용자에게 경고:
  ```
  ⚠ 범위 위반 감지
  허용 범위 외 수정된 파일:
    - [file] — [변경 내용]
  ```
- **AskUserQuestion**:
  Question: "수정 범위를 벗어난 변경이 있습니다. 어떻게 할까요?"
  Header: "Scope"
  Options:
  - **허용** - "이 파일도 수정 범위에 포함"
  - **되돌리기** - "범위 외 변경만 되돌리기"
  - **전체 되돌리기** - "모든 수정을 되돌리고 재시도"

## Phase 4.5: 근본 원인 검증 (Root Cause Verification)

수정이 증상만 가리는 게 아닌지 검증합니다.

서브에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"opus"**):

> **시니어 프로덕트 매니저**로서 버그 수정의 적절성을 검증합니다.
>
> **원래 버그**: [$ARGUMENTS]
> **근본 원인**: [Phase 2 결과]
> **수정 내용**: [Phase 4에서 Developer가 반환한 변경 요약]
> **수정된 파일**: [수정된 파일 목록 — Read로 읽어서 확인]
> **영향 분석**: [Phase 3 결과]
>
> 다음 질문에 답합니다:
>
> 1. **근본 원인 해결 여부**: 이 수정이 근본 원인을 직접 해결하는가, 아니면 증상만 우회하는가?
>    - 예: null 체크 추가는 증상 우회. null이 들어오는 원인을 고치는 것이 근본 해결.
> 2. **완전성**: 같은 근본 원인에서 파생될 수 있는 다른 증상이 있는가?
> 3. **새 버그 가능성**: 이 수정이 새로운 버그를 만들 가능성이 있는가?
>    - 버그 의존 코드가 모두 처리되었는지 확인
> 4. **부작용**: 수정이 기존 동작을 의도치 않게 변경하는가?
>
> 판정:
> ```
> ## 근본 원인 검증
>
> **결과**: PASS | WARN | FAIL
>
> ### 근본 원인 해결
> [해결됨 / 증상만 우회 — 설명]
>
> ### 완전성
> [다른 파생 증상 유무]
>
> ### 새 버그 가능성
> [없음 / 있음 — 구체적 위험]
>
> ### 부작용
> [없음 / 있음 — 설명]
>
> ### 추가 수정 필요 (WARN/FAIL인 경우)
> - [구체적 수정 사항]
> ```

### 검증 결과 처리

**PASS**: Phase 5로 진행.

**WARN**: 사용자에게 경고를 표시하고 진행 여부 확인:
- **AskUserQuestion**:
  Question: "수정에 주의사항이 있습니다. 계속할까요?"
  Header: "Verify"
  Options:
  - **진행** - "경고를 인지하고 계속"
  - **보완** - "추가 수정 사항 적용 후 진행"
  - **중단** - "수정을 되돌리고 수동 처리"

**FAIL**:
- 사용자에게 실패 내용 표시
- Developer 에이전트 재실행 (검증 피드백 포함, Phase 4a로 돌아감)
- **최대 2회 재시도**, 초과 시 사용자에게 수동 처리 안내

## Phase 5: 변경사항 확인

git 저장소인 경우, 수정 내용을 사용자에게 보여주고 확인받습니다.

1. Bash로 `git diff --stat` 실행하여 변경 요약 표시
2. Bash로 `git diff` 실행하여 상세 diff 표시

사용자에게 표시:
```
버그 수정 완료 — 변경사항 확인
──────────────────────────────────────
[git diff --stat 출력]
```

**AskUserQuestion**으로 확인:

Question: "수정 결과를 적용할까요?"
Header: "Confirm"
Options:
- **적용** - "변경사항을 유지하고 다음 단계로 진행"
- **되돌리기** - "모든 변경사항을 되돌리고 중단"
- **부분 되돌리기** - "특정 파일만 되돌리기"

**적용** 선택 시: Phase 6으로 진행.

**되돌리기** 선택 시:
- `git checkout -- .` 실행
- 새로 생성된 파일은 수동으로 삭제 확인
- "변경사항이 되돌려졌습니다." 메시지 출력 후 중단

**부분 되돌리기** 선택 시:
- 변경된 파일 목록을 보여주고 되돌릴 파일을 AskUserQuestion (multiSelect)으로 선택
- 선택된 파일에 대해 `git checkout -- [file]` 실행
- Phase 6으로 진행

## Phase 6: 회귀 테스트

### 6a. 기존 테스트 실행

프로젝트에 테스트 러너가 있으면 (package.json scripts, Makefile, go test 등):

1. Bash로 기존 테스트 스위트 전체 실행
2. 결과 캡처

**테스트 실패 시:**
- 실패한 테스트 목록을 사용자에게 표시
- **AskUserQuestion**:
  Question: "기존 테스트가 실패했습니다. 어떻게 할까요?"
  Header: "Tests"
  Options:
  - **수정** - "실패한 테스트가 버그 수정을 반영하도록 업데이트"
  - **조사** - "테스트 실패가 회귀인지 확인"
  - **스킵** - "테스트 실패를 무시하고 진행"

  **수정** 선택 시: 실패한 테스트가 버그의 (잘못된) 이전 동작을 테스트하는 경우 → 테스트를 올바른 동작으로 업데이트
  **조사** 선택 시: 실패 원인을 분석하여 회귀인지 의도된 변경인지 판단 → 결과에 따라 처리

### 6b. 회귀 테스트 생성

QA 에이전트 실행 (Task tool, subagent_type: "general-purpose", model: **"sonnet"**):

> **시니어 QA 엔지니어**로서 버그 수정에 대한 회귀 테스트를 작성합니다.
>
> **원래 버그**: [$ARGUMENTS — 증상 설명]
> **근본 원인**: [Phase 2 결과]
> **수정된 파일**: [Phase 4에서 수정된 파일 목록]
> **수정 내용**: [변경 요약]
> **테스트 위치**: [.crew/config.md의 test_dir 설정]
> **기존 테스트 패턴**: [프로젝트의 테스트 프레임워크와 패턴]
>
> 작성할 테스트:
> 1. **버그 재현 테스트**: 수정 전이었다면 실패했을 테스트 케이스 (수정 후 통과)
> 2. **엣지 케이스 테스트**: 같은 근본 원인의 다른 트리거 조건
> 3. **영향 범위 테스트**: 수정이 기존 동작을 깨뜨리지 않는지 확인
>
> 기존 테스트 컨벤션을 정확히 따릅니다.
> 반환: 생성된 테스트 파일, 각 테스트 케이스 설명

QA 에이전트 반환 후:
- 생성된 테스트를 실행하여 통과 여부 확인
- 실패하면 수정 (최대 2회 재시도)

## Phase 7: 마무리

### 7a. README.md 동기화

프로젝트 루트에 `README.md`가 존재하는지 Glob으로 확인합니다.

**`README.md`가 존재하면:**

이번 버그 수정에서 변경된 파일 목록을 기반으로, README.md에 영향을 줄 수 있는 변경이 있는지 판단합니다:
- API 동작 변경 (파라미터, 응답, 에러 코드 등)
- 설정/환경변수 변경
- 사용법 또는 제약사항 변경
- 알려진 이슈/제한사항 업데이트

**영향이 있으면** `README.md`를 Read로 읽고, 변경사항에 맞춰 해당 섹션을 Edit으로 업데이트합니다. 기존 README.md의 스타일과 구조를 유지합니다.

**`README.md`가 없거나 영향이 없으면:** 이 단계를 스킵합니다.

### 7b. 아티팩트 저장

버그명에서 slug 생성 (예: "null-pointer-auth" 등).

디버그 리포트를 `.crew/artifacts/review/debug-[slug].md`에 저장합니다. 다음 섹션을 포함:
- **근본 원인**: 유형, 위치(file:line), 설명, 도입 시점
- **영향 분석**: 위험 등급, 직접 영향 파일 수, 버그 의존 코드 유무
- **수정 내용**: 수정 파일, 변경 요약, 근본 원인 검증 결과
- **회귀 테스트**: 생성 테스트 파일, 기존 테스트 결과
- **교훈**: 같은 유형 버그 예방을 위한 제안

### 7c. 보드 업데이트

`.crew/config.md`에서 `last_task_id`를 읽고 증가시킵니다.

`.crew/board.md`의 `## Done` 섹션에 태스크를 추가합니다:

```markdown
### TASK-[NNN]: 🐛 [버그 요약]
- **Role**: Developer
- **Priority**: P0
- **Depends on**: none
- **Feature**: fix-[slug]
- **Files**: [수정된 파일 목록]
- **Description**: [근본 원인 한 줄 요약]
- **Resolution**: [수정 내용 한 줄 요약]
```

`> Last updated:` 타임스탬프 업데이트.

### 7d. 최종 요약

최종 요약을 제시합니다: 버그 증상, 근본 원인 (유형/위치/설명), 영향 분석 (위험 등급/영향 파일 수/버그 의존 코드), 수정 내용 (파일/검증 결과), 테스트 (회귀 테스트/기존 테스트), 리포트 경로, 태스크 ID.

## Phase 8: CLAUDE.md 상태 업데이트

`CLAUDE.md`의 `## Crew 현재 상태` 섹션을 업데이트합니다 (없으면 파일 끝에 추가, 있으면 Edit으로 교체). `.crew/board.md`를 읽어 다음을 포함:
- 마지막 업데이트 타임스탬프
- 진행 중인 작업 (In Progress/In Review 태스크)
- 활성 스프린트 정보
- 이번 실행에서 생성된 아티팩트 경로 (디버그 리포트)
- 다음 명령 제안
